⚠️ Commands requiring extra caution

DROP USER ... CASCADEDeletes the user AND all their tables, indexes, procedures — no warning
SHUTDOWN ABORTUngraceful stop — DB will need instance recovery on next STARTUP
ALTER TABLE ... DROP COLUMNColumn drop is not easily reversible — backup/confirm first
ALTER SYSTEM KILL SESSIONKills any session including DBA sessions — always confirm the SID first
TRUNCATE TABLE schema.tableRemoves ALL rows instantly — no rollback. Very different from DELETE.
ALTER DATABASE BACKUP CONTROLFILE TO TRACEUseful but run only as needed; can conflict with open RMAN sessions

RMAN quick tips
Test before productionAlways run RESTORE ... PREVIEW first to confirm RMAN has the backups it needs before starting recovery.
Know your modeFull recovery needs the DB MOUNTED (not open). Use SHUTDOWN IMMEDIATE then STARTUP MOUNT before restoring.
Channel = pipeA channel is a connection from RMAN to the OS. More channels = parallel = faster. Match channels to CPU/disk count.
CROSSCHECK firstBefore any recovery, run CROSSCHECK BACKUP to make sure RMAN knows which files actually exist on disk.

1

Situation Assessment

Database is down, corrupted, or experiencing datafile loss. RMAN backups are available. Recovery from full backup is required.

Access to Oracle server as oracle OS user. RMAN backups accessible (disk or tape). ORACLE_SID and ORACLE_HOME set correctly.

HIGH — this procedure involves full database outage and potential data loss up to the point of the last backup + applied archivelogs.

🚨 Stop and confirm before proceeding

Confirm with your team lead or DBA manager before starting. Identify the target recovery point (current time, specific SCN, or specific timestamp). Ensure all users are notified of the outage. If uncertain — escalate rather than proceed.

2

Pre-Recovery Checklist

  • Confirm Oracle environment variables are correct
    echo $ORACLE_SID $ORACLE_HOME $PATH — must match the failed database
    Critical
  • Verify RMAN backup files are accessible and not corrupted
    Check backup destination directory: ls -lh /backup/rman/ — confirm backup files exist and are readable
    Critical
  • Run CROSSCHECK to verify RMAN catalog is aware of available backups
    CROSSCHECK BACKUP; CROSSCHECK ARCHIVELOG ALL;
    Critical
  • Identify the target recovery SCN or timestamp
    For current recovery: bring to latest archivelog. For PITR: confirm exact timestamp/SCN with the application team.
    Required
  • Confirm sufficient disk space at restore destination
    df -h /u01/oradata — must have >= the size of all datafiles to restore
    Critical
  • Notify all application teams and stakeholders of the outage
    Manual
  • Document the incident: time of failure, symptoms, and cause if known
    Manual
3

Recovery Steps

⚠️ Complete each step before moving to the next

Do not skip steps. Record the output of each command. If any step produces unexpected errors, stop and escalate to your senior DBA before continuing.

#ActionCommand / DetailOwnerType
1
Set OS environment variables
Confirm ORACLE_SID matches the database you want to recover.
export ORACLE_SID=ORCL
export ORACLE_HOME=/u01/app/oracle/product/19c/dbhome_1
export PATH=$ORACLE_HOME/bin:$PATH
 DBA Manual
2
Shutdown the database (if still running)
Use ABORT only if IMMEDIATE hangs — this is normal in a crashed state.
sqlplus / as sysdba
SHUTDOWN ABORT;
 DBA Manual
3
Start the database in MOUNT state
MOUNT state allows RMAN to operate without opening the database to users.
STARTUP MOUNT;
 DBA Manual
4
Connect to RMAN and crosscheck backups
Ensures RMAN knows about all available backup pieces before starting restore.
rman target /
CROSSCHECK BACKUP;
CROSSCHECK ARCHIVELOG ALL;
DELETE EXPIRED BACKUP;
DELETE EXPIRED ARCHIVELOG ALL;
 DBA Critical
5
Preview the restore (DRY RUN — no changes)
PREVIEW shows exactly which backupsets RMAN will use. Confirm they exist and are accessible.
RESTORE DATABASE PREVIEW SUMMARY;
 DBA Critical
6
Restore the database datafiles
RMAN reads backupsets and copies datafiles to their original locations. This may take 30 minutes to several hours depending on DB size.
RESTORE DATABASE;
 DBA Critical
7a
Recover to current time (full recovery)
Applies all available archivelogs to bring database to most recent consistent state. Use 7a OR 7b — not both.
RECOVER DATABASE;
 DBA Full Rec.
7b
Recover to a point in time (PITR)
Use when recovering to before a bad transaction. Confirm target time with app team. RESETLOGS required.
RECOVER DATABASE UNTIL TIME
"TO_DATE('2025-11-01 02:30:00','YYYY-MM-DD HH24:MI:SS')";
 DBA PITR
8
Open the database
Use RESETLOGS if you used PITR (step 7b) or incomplete recovery. Use normal OPEN for complete recovery.
-- After PITR:
ALTER DATABASE OPEN RESETLOGS;
-- After complete recovery:
ALTER DATABASE OPEN;
 DBA Critical
9
Validate DB is open and healthy
Confirm STATUS=OPEN, OPEN_MODE=READ WRITE. Check alert log for any ORA- errors.
SELECT NAME, OPEN_MODE FROM V$DATABASE;
SELECT STATUS FROM V$INSTANCE;
-- Check alert log for errors
 DBA Critical
10
Take a fresh RMAN backup immediately
After RESETLOGS, old backups cannot be used. Always take a fresh full backup before declaring recovery complete.
BACKUP DATABASE PLUS ARCHIVELOG DELETE INPUT;
 DBA Critical
11
Notify stakeholders — DB is back online
Post recovery summary: duration, recovery point achieved, data loss (if any), and next steps.
Slack / Email → notify app teams & management
 DBA Manual
4

If Recovery Fails

🚨 Escalate immediately if these occur

RMAN-6026 (no backup available), RMAN-20201 (archivelog not found), or ORA-01547 (failed to open database) — stop and escalate to senior DBA.

🔄 Troubleshooting steps

  • Re-run CROSSCHECK — backup file may have moved
  • CATALOG missing backup files manually
    CATALOG START WITH '/backup/';
  • For missing archivelogs: recover with ALLOWLOGS
    RECOVER DATABASE ALLOW 1 CORRUPTION;
  • Check alert log for ORA- errors — these diagnose root cause
  • Escalate to Oracle Support (SR) if ORA-600 or ORA-7445 appears
5

Post-Recovery Checklist

✅ Complete within 24 hours of recovery

Document everything. A recovery that isn't documented is a recovery waiting to fail again.

  • Confirm all tablespaces are ONLINE and READ WRITE
    SELECT TABLESPACE_NAME, STATUS FROM DBA_TABLESPACES;
  • Recompile any invalid objects after recovery
    EXEC UTL_RECOMP.recomp_serial();
  • Fresh full RMAN backup completed successfully
  • Application smoke tests passed
  • Incident documented with RCA and timeline
  • Recovery point achieved and data loss (if any) confirmed to management

Start with waitsDon't guess — let Oracle tell you what it's waiting for. V$SESSION_WAIT and V$SYSTEM_EVENT show the truth.

Overview & Scope

Applying Oracle quarterly Release Updates (RU) to a non-RAC Oracle 19c/21c database — binary patching + datapatch SQL component.

Scheduled quarterly patching window. Also use for applying Critical Patch Updates (CPU) when security advisory mandates it.

Junior DBA (with DBA Lead oversight for first few times). Requires OS oracle user access and SYSDBA database privileges.

🚨 Do NOT proceed if any of these are true

• Active RMAN backup is running · • Any critical jobs are in progress · • DB is not in ARCHIVELOG mode · • No full backup taken in last 24 hours · • SYSTEM/SYSAUX tablespaces are more than 90% full · • Previous patch is in a failed/pending state (check dba_registry_sqlpatch)

🟢 Start and Shut Down an Oracle Database

The first task every DBA needs to master — safely bringing a database online and offline

Skill Level
Beginner
Time Required
10 min
Category
Database Administration
Prerequisites
Oracle installed · oracle OS user · ORACLE_SID set
Applies To
Oracle 19c / 21c
1

What You Will Learn

Starting and stopping a database properly protects your data. Oracle has multiple shutdown modes — using the wrong one can corrupt a database. This guide shows you the correct sequence and what to verify at each step.
2

Before You Begin

  • Be logged in as the oracle OS user
    su - oracle — never run sqlplus as root
  • ORACLE_SID and ORACLE_HOME are set
    echo $ORACLE_SID — should show your database name
  • You have SYSDBA privilege
    sqlplus / as sysdba — this uses OS authentication, no password needed
3

Step-by-Step Instructions

#What to DoCommandExpected Result
1
Connect to the database as SYSDBA
 sqlplus / as sysdba
SQL> prompt appears. If you get ORA-01034 the instance is already down.
2
Check the current database status
 SELECT STATUS FROM V$INSTANCE;
Returns STARTED, MOUNTED, or OPEN. OPEN = fully running.
3
Start the database (if it is down)
 STARTUP;
Three lines appear: ORACLE instance started → Database mounted → Database opened. All three must show.
4
Verify the database is open and healthy
 SELECT NAME, OPEN_MODE, DB_UNIQUE_NAME FROM V$DATABASE;
OPEN_MODE should be READ WRITE. Any other value needs investigation.
5
Confirm all datafiles are online
 SELECT FILE#, STATUS, NAME FROM V$DATAFILE WHERE STATUS != 'ONLINE';
Zero rows = all datafiles online. Any rows here = problem.
6
Start the Listener (if not running)
 EXIT;
lsnrctl start
You must exit sqlplus first. Listener starts and reports "The command completed successfully".
7
Test connectivity from the server
 tnsping ORCL
Should return "OK (X msec)". This confirms applications can reach the database.
8
--- SHUTDOWN STEPS --- Check for active sessions before stopping
 SELECT COUNT(*) FROM V\$SESSION WHERE TYPE = 'USER';
Note the number. If > 0 notify users before shutdown.
9
Shut down the database cleanly
 STARTUP;
sqlplus / as sysdba
SHUTDOWN IMMEDIATE;
IMMEDIATE waits for current SQL to finish, rolls back uncommitted transactions, then shuts down. This is the standard method.
10
Verify shutdown completed
 SELECT STATUS FROM V$INSTANCE;
Should return ORA-01034: ORACLE not available — this is expected and means it is down.
11
Stop the Listener after shutdown
 EXIT;
lsnrctl stop
Prevents applications from attempting new connections while DB is offline.
4

Common Errors & Fixes

Error / SymptomWhat to Do
ORA-01031: insufficient privilegesYou are not connecting as SYSDBA. Use: sqlplus / as sysdba
ORA-01034: ORACLE not availableInstance is down. Run STARTUP; to bring it up.
ORA-00119 / ORA-00132Parameter file (spfile/pfile) issue. Check $ORACLE_HOME/dbs/ for initSID.ora or spfileSID.ora

👤 Create a User and Grant Privileges

Set up a new database user with the right permissions — a daily DBA task

Skill Level
Beginner
Time Required
15 min
Category
Database Administration
Prerequisites
Database is OPEN · Connected as SYSDBA or user with CREATE USER privilege
Applies To
Oracle 19c / 21c (CDB and non-CDB)
1

What You Will Learn

Creating a user involves three things: the user account, a default tablespace for their data, and the correct privileges. Too many privileges = security risk. Too few = user cannot work. This guide walks through the right approach.
2

Before You Begin

  • Database is open
    SELECT OPEN_MODE FROM V\$DATABASE; — must show READ WRITE
  • Know the intended tablespace for the user
    SELECT TABLESPACE_NAME FROM DBA_TABLESPACES; — pick one that isn't SYSTEM or SYSAUX
  • Have a naming convention ready
    Example: APP_OWNER for schema owner, APP_RO for read-only user
3

Step-by-Step Instructions

#What to DoCommandExpected Result
1
Check if the user already exists
 SELECT USERNAME, ACCOUNT_STATUS FROM DBA_USERS
WHERE USERNAME = 'MYUSER';
Zero rows = safe to create. If user exists, check ACCOUNT_STATUS before recreating.
2
Check available tablespaces
 SELECT TABLESPACE_NAME, STATUS,
ROUND(SUM(BYTES)/1024/1024,1) MB
FROM DBA_DATA_FILES
GROUP BY TABLESPACE_NAME, STATUS;
Pick a tablespace for the user — NOT SYSTEM, NOT SYSAUX. Use USERS or a custom app tablespace.
3
Create the user with a default tablespace
 CREATE USER myuser
IDENTIFIED BY "SecurePass123!"
DEFAULT TABLESPACE users
TEMPORARY TABLESPACE temp
QUOTA 500M ON users;
User created. The QUOTA limits how much space they can consume — important to set!
4
Grant CREATE SESSION so they can log in
 GRANT CREATE SESSION TO myuser;
Without this the user cannot connect at all. Minimum required grant.
5
Grant object privileges for a schema owner
 GRANT CREATE TABLE, CREATE VIEW,
CREATE SEQUENCE, CREATE PROCEDURE,
CREATE TRIGGER, CREATE INDEX
TO myuser;
These are typical privileges for an application schema owner. Only grant what is needed.
6
OR — grant read-only access to another schema
 GRANT SELECT ON hr.employees TO myuser;
GRANT SELECT ON hr.departments TO myuser;
-- Or grant all tables at once:
BEGIN
FOR t IN (SELECT table_name FROM dba_tables WHERE owner='HR') LOOP
EXECUTE IMMEDIATE 'GRANT SELECT ON hr.'||t.table_name||' TO myuser';
END LOOP;
END;
/
Read-only users only need SELECT. Never give DBA or CONNECT unless truly necessary.
7
Verify the user was created correctly
 SELECT USERNAME, DEFAULT_TABLESPACE,
TEMPORARY_TABLESPACE, ACCOUNT_STATUS
FROM DBA_USERS
WHERE USERNAME = 'MYUSER';
ACCOUNT_STATUS should be OPEN. DEFAULT_TABLESPACE should be what you set.
8
Verify their privileges
 SELECT PRIVILEGE FROM DBA_SYS_PRIVS
WHERE GRANTEE = 'MYUSER'
ORDER BY PRIVILEGE;
Lists all system privileges granted. Should match exactly what you granted.
9
Test by connecting as the new user
 CONNECT myuser/SecurePass123!
SELECT USER FROM DUAL;
Should return MYUSER. Connection works = user is set up correctly.
4

Common Errors & Fixes

Error / SymptomWhat to Do
ORA-01920: user name conflicts with another userUser already exists. Use ALTER USER instead, or drop and recreate.
ORA-00959: tablespace does not existThe tablespace name is wrong. Run SELECT TABLESPACE_NAME FROM DBA_TABLESPACES; to get valid names.
ORA-01045: user lacks CREATE SESSION privilegeYou forgot to GRANT CREATE SESSION. Run it now.
ORA-01536: space quota exceededThe user hit their quota limit. Run: ALTER USER myuser QUOTA UNLIMITED ON users;

🗂️ Create and Manage a Tablespace

Understand how Oracle stores data and how to add storage when needed

Skill Level
Beginner
Time Required
20 min
Category
Database Administration
Prerequisites
DB is OPEN · Connected as SYSDBA · Know where your datafiles are stored
Applies To
Oracle 19c / 21c
1

What You Will Learn

A tablespace is Oracle's logical storage container. Every table, index, and object lives in a tablespace, which maps to one or more physical datafiles on disk. When tablespaces fill up, the database stops accepting writes — knowing how to monitor and extend them is critical.
2

Before You Begin

  • Find where existing datafiles live
    SELECT NAME FROM V\$DATAFILE FETCH FIRST 3 ROWS ONLY; — copy the directory path
  • Check available disk space
    df -h /u01 — ensure you have space for new datafiles
  • Know the name for your new tablespace
    Use a clear naming convention: APP_DATA, REPORTING_DATA, etc.
3

Step-by-Step Instructions

#What to DoCommandExpected Result
1
Check existing tablespaces and their usage
 SELECT t.tablespace_name,
ROUND(SUM(d.bytes)/1024/1024,0) total_mb,
ROUND(SUM(d.bytes - NVL(f.bytes,0))/1024/1024,0) used_mb,
ROUND(NVL(SUM(f.bytes),0)/1024/1024,0) free_mb
FROM dba_tablespaces t
JOIN dba_data_files d ON t.tablespace_name = d.tablespace_name
LEFT JOIN (SELECT tablespace_name, SUM(bytes) bytes FROM dba_free_space GROUP BY tablespace_name) f
ON t.tablespace_name = f.tablespace_name
GROUP BY t.tablespace_name
ORDER BY used_mb DESC;
Shows all tablespaces with total, used, and free MB. Any tablespace over 85% full needs attention.
2
Find where datafiles are stored on disk
 SELECT TABLESPACE_NAME, FILE_NAME
FROM DBA_DATA_FILES
WHERE TABLESPACE_NAME = 'USERS';
Note the directory path — use the same path for your new datafile.
3
Create a new tablespace
 CREATE TABLESPACE app_data
DATAFILE '/u01/app/oracle/oradata/ORCL/app_data01.dbf'
SIZE 500M
AUTOEXTEND ON NEXT 100M MAXSIZE 5G
EXTENT MANAGEMENT LOCAL
SEGMENT SPACE MANAGEMENT AUTO;
Tablespace created. AUTOEXTEND ON means it grows automatically up to MAXSIZE — always set a MAXSIZE to prevent runaway growth.
4
Verify the new tablespace exists
 SELECT TABLESPACE_NAME, STATUS,
EXTENT_MANAGEMENT, SEGMENT_SPACE_MANAGEMENT
FROM DBA_TABLESPACES
WHERE TABLESPACE_NAME = 'APP_DATA';
STATUS = ONLINE, EXTENT_MANAGEMENT = LOCAL, SEGMENT_SPACE_MANAGEMENT = AUTO — this is the correct modern setup.
5
Add a datafile when a tablespace is getting full
 ALTER TABLESPACE app_data
ADD DATAFILE '/u01/app/oracle/oradata/ORCL/app_data02.dbf'
SIZE 500M
AUTOEXTEND ON NEXT 100M MAXSIZE 5G;
This is the most common way to extend a tablespace. Adding a datafile does not require downtime.
6
Resize an existing datafile
 ALTER DATABASE DATAFILE
'/u01/app/oracle/oradata/ORCL/app_data01.dbf'
RESIZE 2G;
You can increase (or decrease if free space exists) the size of a datafile directly.
7
Enable autoextend on an existing datafile
 ALTER DATABASE DATAFILE
'/u01/app/oracle/oradata/ORCL/app_data01.dbf'
AUTOEXTEND ON NEXT 100M MAXSIZE 10G;
Turn on autoextend for a datafile that was created without it. Always set a MAXSIZE.
8
Set the new tablespace as default for a user
 ALTER USER myuser DEFAULT TABLESPACE app_data
QUOTA UNLIMITED ON app_data;
New objects created by this user will go into APP_DATA by default.
9
Drop a tablespace (when no longer needed)
 DROP TABLESPACE app_data
INCLUDING CONTENTS AND DATAFILES;
INCLUDING CONTENTS removes all objects inside. INCLUDING CONTENTS AND DATAFILES also deletes the physical file. NEVER drop SYSTEM, SYSAUX, UNDO, or TEMP.
4

Common Errors & Fixes

Error / SymptomWhat to Do
ORA-01119: error in creating database fileDirectory does not exist or oracle user has no write permission. Check: ls -ld /u01/app/oracle/oradata/ORCL/
ORA-01144: File size exceeds maximumYour OS block size limits file size. Use BIGFILE tablespace or add a second datafile instead.
Tablespace at 100% — writes failingImmediate fix: ALTER TABLESPACE app_data ADD DATAFILE ... — do not wait for this, data writes fail when tablespace is full.
ORA-00059: maximum number of datafiles exceededHit the DB_FILES parameter limit. Increase it: ALTER SYSTEM SET DB_FILES=500 SCOPE=SPFILE; then restart.

🔒 Diagnose and Resolve Locking Sessions

Find what is blocking whom — and safely release the lock

Skill Level
Intermediate
Time Required
15 min
Category
Database Administration
Prerequisites
DB is OPEN · Connected as SYSDBA · Access to V$SESSION and V$LOCK views
Applies To
Oracle 19c / 21c
1

What You Will Learn

Locking is normal in Oracle — it protects data consistency. But when a session holds a lock and does not commit, other sessions queue up waiting. Users report "the application is hanging". This guide shows how to find the blocker, understand why it is stuck, and resolve it safely.
2

Before You Begin

  • Get the user or app team to tell you what is hanging
    Knowing the application, table, or user helps narrow down the search
  • Note the approximate start time of the problem
    V$SESSION has LOGON_TIME and LAST_CALL_ET to help you identify old sessions
  • Understand that killing a session has consequences
    The blocked session's transaction will be rolled back — data is safe but the user will get an error
3

Step-by-Step Instructions

#What to DoCommandExpected Result
1
Find blocking and blocked sessions
 SELECT
blocker.sid blocker_sid,
blocker.serial# blocker_serial,
blocker.username blocker_user,
blocker.status blocker_status,
blocker.program blocker_program,
waiter.sid waiter_sid,
waiter.username waiter_user,
waiter.wait_class wait_class,
obj.object_name locked_object
FROM v\$session blocker
JOIN v\$lock lk1 ON blocker.sid = lk1.sid AND lk1.block = 1
JOIN v\$lock lk2 ON lk1.id1 = lk2.id1 AND lk1.id2 = lk2.id2
JOIN v\$session waiter ON waiter.sid = lk2.sid AND lk2.request > 0
LEFT JOIN dba_objects obj ON lk1.id1 = obj.object_id
ORDER BY blocker_sid;
Shows the BLOCKER (who holds the lock) and the WAITER (who is stuck). Focus on the BLOCKER — that is who you need to investigate.
2
Get more detail about the blocking session
 SELECT SID, SERIAL#, USERNAME, STATUS,
OSUSER, MACHINE, PROGRAM,
MODULE, ACTION,
TO_CHAR(LOGON_TIME,'DD-MON HH24:MI') logon,
LAST_CALL_ET seconds_idle
FROM V\$SESSION
WHERE SID = &blocker_sid;
LAST_CALL_ET = seconds since the session last did anything. If it is thousands of seconds, the user likely walked away from an open transaction.
3
Find the SQL the blocking session last ran
 SELECT sql_text
FROM v\$sqlarea
WHERE sql_id = (
SELECT sql_id FROM v\$session WHERE sid = &blocker_sid
);
This shows the SQL that started the lock. Gives you context before deciding whether to kill.
4
Count how many sessions are waiting (queue depth)
 SELECT COUNT(*) waiting_sessions
FROM V\$SESSION
WHERE BLOCKING_SESSION = &blocker_sid;
If this number is large and growing, the situation is urgent.
5
Try asking the user to commit or rollback
 Contact the user or application identified in PROGRAM/MACHINE
Ask them to commit or rollback their transaction
Best option — let the user resolve it. Avoids the risk of data loss from a forced kill.
6
If the user is unreachable — kill the blocking session
 ALTER SYSTEM KILL SESSION '&blocker_sid,&blocker_serial' IMMEDIATE;
IMMEDIATE forces the kill. Oracle will roll back the blocker's uncommitted transaction automatically. Blocked sessions will resume.
7
Verify the block is resolved
 SELECT SID, STATUS, USERNAME FROM V\$SESSION
WHERE SID = &blocker_sid;
The session should be gone (no rows) or STATUS = KILLED. Blocked sessions should now show STATUS = ACTIVE.
8
Check alert log for any errors after kill
 -- From OS terminal:
tail -20 $ORACLE_BASE/diag/rdbms/orcl/ORCL/trace/alert_ORCL.log
A killed session generates a brief ORA-00031 entry which is normal. Look for any other ORA- errors.
4

Common Errors & Fixes

Error / SymptomWhat to Do
ALTER SYSTEM KILL SESSION does not work immediatelyThe session may be in the middle of a rollback. Check: SELECT STATUS FROM V$SESSION WHERE SID=&sid — if STATUS=KILLED it will clean up shortly. Wait it out.
ORA-00030: user session ID does not existSession already disconnected by the time you ran the kill — the lock is already released.
Lock keeps coming backApplication is re-opening the same transaction. This is a code or connection pool issue — escalate to the app team.
Cannot identify the blocker usernameOSUSER and MACHINE columns help identify the actual person. PROGRAM shows the application (JDBC Thin Client, sqlplus, etc.).

The alert log is your database diary — learn to read it fast

Skill Level
Beginner
Time Required
10 min
Category
Database Administration
Prerequisites
Oracle OS user access · DB running (or alert log accessible)
Applies To
Oracle 19c / 21c
1

What You Will Learn

The alert log records everything important that happens to a database — startups, shutdowns, errors, parameter changes, and more. It is always the first place to look when something goes wrong. This guide shows you where to find it, how to read it, and what to look for.
2

Before You Begin

  • Know your ORACLE_BASE
    echo $ORACLE_BASE — usually /u01/app/oracle
  • Know your ORACLE_SID
    echo $ORACLE_SID — the name of your database instance
3

Step-by-Step Instructions

#What to DoCommandExpected Result
1
Find the alert log location from inside the database
 SELECT VALUE FROM V\$DIAG_INFO
WHERE NAME = 'Diag Trace';
Returns the directory path. The alert log is named alert_<SID>.log in that directory.
2
Navigate to the alert log directory
 cd $ORACLE_BASE/diag/rdbms/$(echo $ORACLE_SID | tr A-Z a-z)/$ORACLE_SID/trace/
ls -lh alert_${ORACLE_SID}.log
You should see the alert log file. Note its size — a very large file means it has never been rotated.
3
View the last 50 lines (most recent activity)
 tail -50 alert_${ORACLE_SID}.log
Most recent entries appear at the bottom. Each entry has a timestamp. Look for ORA- error codes.
4
Live-monitor the alert log (useful during startups/patching)
 tail -f alert_${ORACLE_SID}.log
Streams new lines as they are written. Press Ctrl+C to stop. Use this during any maintenance activity.
5
Search for all ORA- errors today
 grep -i 'ORA-' alert_${ORACLE_SID}.log | grep $(date +%Y-%m-%d)
Shows only today's errors. If no date match, try the format in your log: grep "$(date +"%a %b")" alert*.log
6
Search for errors in a specific time window
 awk '/Fri Apr 04/{found=1} found{print}' alert_${ORACLE_SID}.log | head -100
Change the date string to match your timezone format in the log. Shows everything from that point.
7
Count and rank the most frequent errors
 grep 'ORA-' alert_${ORACLE_SID}.log | grep -oP 'ORA-[0-9]+' | sort | uniq -c | sort -rn | head -20
Shows which error codes appear most often — helps you prioritize what to investigate.
8
Use adrci for a cleaner view
 adrci
adrci> show alert -tail 100
adrci> show alert -p "message_text like '%ORA-%'"
adrci> quit
adrci is Oracle's Automatic Diagnostic Repository command-line tool. Cleaner than grep for structured queries.
9
Understand the key log entries to watch for
 -- Normal entries (safe to ignore):
-- Thread 1 advanced to log sequence ...
-- Completed checkpoint ...
-- Starting background process ...

-- Investigate immediately:
-- ORA-00600 (internal error)
-- ORA-07445 (process crash)
-- ORA-01555 (snapshot too old)
-- ORA-04031 (shared memory error)
-- ORA-27063 / ORA-01144 (I/O errors)
Knowing which errors are routine vs critical is a key DBA skill.
4

Common Errors & Fixes

Error / SymptomWhat to Do
Alert log file is enormous (hundreds of MB)Safe to gzip old logs: gzip alert_ORCL.log.bak. Oracle rotates on restart. Use adrci to purge: adrci> purge -age 43200
Cannot find alert log with expected pathTry: find $ORACLE_BASE -name "alert_*.log" 2>/dev/null
Log shows ORA-00600 or ORA-07445These are internal errors — search the error number and incident ID on My Oracle Support (support.oracle.com) immediately.
No errors but application is complainingCheck trace files: ls -lt $ORACLE_BASE/diag/.../trace/*.trc | head — the newest .trc file has session-level errors.

💾 Configure RMAN and Run Your First Backup

Set up Oracle's built-in backup tool and take a complete database backup

Skill Level
Beginner
Time Required
30–45 min
Category
RMAN Backup & Recovery
Prerequisites
DB in ARCHIVELOG mode · Sufficient disk space · oracle OS user
Applies To
Oracle 19c / 21c
1

What You Will Learn

RMAN (Recovery Manager) is Oracle's built-in backup and recovery tool. It is the only supported way to take online backups of an Oracle database. Before your first backup you need to configure where backups go and how long to keep them. This guide covers initial setup and running a full backup.
2

Before You Begin

  • Verify the database is in ARCHIVELOG mode
    SELECT LOG_MODE FROM V\$DATABASE; — must show ARCHIVELOG. If it says NOARCHIVELOG, you cannot do online backups.
  • Check available disk space
    df -h — you need roughly 1x the database size for a compressed backup
  • Create the backup destination directory
    mkdir -p /backup/rman && chown oracle:oinstall /backup/rman
3

Step-by-Step Instructions

#What to DoCommandExpected Result
1
Connect to RMAN
 rman target /
RMAN> prompt appears. The / means connect to the local database using OS authentication — no password needed.
2
Check current RMAN configuration
 SHOW ALL;
Lists all RMAN configuration parameters. CONFIGURE statements with DEFAULT are using Oracle defaults.
3
Set the backup destination (Fast Recovery Area)
 -- Option A: File system backup location:
CONFIGURE CHANNEL DEVICE TYPE DISK FORMAT '/backup/rman/%d_%T_%U';

-- Option B: Use Fast Recovery Area (if configured):
-- Already set via DB_RECOVERY_FILE_DEST parameter
The %d = DB name, %T = date, %U = unique ID. These create organized backup filenames.
4
Set backup retention policy
 CONFIGURE RETENTION POLICY TO RECOVERY WINDOW OF 7 DAYS;
Keep backups needed to recover to any point in the last 7 days. Adjust to your Recovery Point Objective (RPO).
5
Enable backup compression
 CONFIGURE COMPRESSION ALGORITHM "MEDIUM" AS OF RELEASE "DEFAULT" OPTIMIZE FOR LOAD TRUE;
CONFIGURE DEVICE TYPE DISK BACKUP TYPE TO COMPRESSED BACKUPSET;
MEDIUM compression reduces backup size by ~50-70% with moderate CPU usage. Use BASIC if CPU is a concern.
6
Set number of backup channels (parallelism)
 CONFIGURE DEVICE TYPE DISK PARALLELISM 2 BACKUP TYPE TO COMPRESSED BACKUPSET;
2 channels = 2 files written in parallel = roughly 2x faster backup. Match to number of CPUs available.
7
Run your first full database backup
 BACKUP DATABASE PLUS ARCHIVELOG DELETE INPUT;
-- This may take 15-60 minutes depending on database size
PLUS ARCHIVELOG backs up archivelogs too. DELETE INPUT removes archivelogs after backing them up to save space.
8
Verify the backup completed successfully
 LIST BACKUP SUMMARY;
Shows all backups. Look for STATUS=A (Available). Any BS key with EXPIRED or UNAVAILABLE needs investigation.
9
Validate the backup is usable
 RESTORE DATABASE VALIDATE;
VALIDATE checks backup integrity WITHOUT actually restoring anything. Safe to run anytime. Should complete with 0 errors.
10
Check backup size and location on disk
 EXIT;
ls -lh /backup/rman/
Verify files exist on disk with expected sizes. Cross-reference with the RMAN LIST output.
4

📅 Set Up an Incremental Backup Strategy

Implement a Level 0 / Level 1 schedule that protects the database efficiently

Skill Level
Intermediate
Time Required
20 min (setup) + backup runtime
Category
RMAN Backup & Recovery
Prerequisites
RMAN configured (Guide 06 complete) · cron access on the server
Applies To
Oracle 19c / 21c
1

What You Will Learn

A full backup every day is slow and expensive. The standard DBA approach is a weekly Level 0 (full) backup and nightly Level 1 incremental backups that capture only changed blocks. This guide sets up that strategy and schedules it with cron.
2

Before You Begin

  • RMAN is configured with a retention policy
    RMAN> SHOW RETENTION POLICY; — should show a configured policy
  • You have a backup destination with enough space
    Full backup + 6 incrementals typically requires ~1.5x your database size
  • You know your maintenance window
    Backups should run during low-activity periods
3

Step-by-Step Instructions

#What to DoCommandExpected Result
1
Understand Level 0 vs Level 1
 -- Level 0 = Full baseline (all blocks):
BACKUP INCREMENTAL LEVEL 0 DATABASE;

-- Level 1 Differential = only blocks changed since last Level 0 or Level 1:
BACKUP INCREMENTAL LEVEL 1 DATABASE;

-- Level 1 Cumulative = blocks changed since last Level 0 ONLY:
BACKUP INCREMENTAL LEVEL 1 CUMULATIVE DATABASE;
Differential is smaller but requires more backups to restore. Cumulative is larger but restores faster. Most shops use differential.
2
Enable block change tracking (makes incrementals faster)
 -- Connect to sqlplus as SYSDBA:
ALTER DATABASE ENABLE BLOCK CHANGE TRACKING
USING FILE '/u01/app/oracle/oradata/ORCL/change_tracking.f';

-- Verify it is enabled:
SELECT STATUS, FILENAME FROM V\$BLOCK_CHANGE_TRACKING;
With change tracking, RMAN only reads changed blocks during incremental — can be 10-20x faster on large DBs.
3
Create the weekly Level 0 backup script
 cat > /scripts/rman_level0.sh << 'EOF'
#!/bin/bash
export ORACLE_SID=ORCL
export ORACLE_HOME=/u01/app/oracle/product/19c/db
export PATH=$ORACLE_HOME/bin:$PATH
LOGFILE=/logs/rman_level0_$(date +%Y%m%d).log

rman target / >> $LOGFILE 2>&1 << RMAN
BACKUP INCREMENTAL LEVEL 0
TAG 'WEEKLY_FULL'
DATABASE PLUS ARCHIVELOG DELETE INPUT;
DELETE NOPROMPT OBSOLETE;
EXIT;
RMAN
EOF
chmod +x /scripts/rman_level0.sh
This script takes the weekly full baseline and deletes old backups that fall outside the retention window.
4
Create the nightly Level 1 backup script
 cat > /scripts/rman_level1.sh << 'EOF'
#!/bin/bash
export ORACLE_SID=ORCL
export ORACLE_HOME=/u01/app/oracle/product/19c/db
export PATH=$ORACLE_HOME/bin:$PATH
LOGFILE=/logs/rman_level1_$(date +%Y%m%d).log

rman target / >> $LOGFILE 2>&1 << RMAN
BACKUP INCREMENTAL LEVEL 1
TAG 'NIGHTLY_INCR'
DATABASE PLUS ARCHIVELOG DELETE INPUT;
EXIT;
RMAN
EOF
chmod +x /scripts/rman_level1.sh
Incremental backups are much smaller and faster — typically 5-15 minutes vs 1 hour for a full.
5
Schedule with cron
 crontab -e
# Add these two lines:
# Level 0 every Sunday at 1 AM:
0 1 * * 0 /scripts/rman_level0.sh
# Level 1 Mon-Sat at 2 AM:
0 2 * * 1-6 /scripts/rman_level1.sh
Verify cron is set: crontab -l — you should see both entries.
6
Test the scripts manually first
 /scripts/rman_level0.sh
echo "Exit code: $?"
Run manually before trusting to cron. Exit code 0 = success. Check the log file for any errors.
7
Add a backup health check to your monitoring
 rman target / << EOF
LIST BACKUP OF DATABASE COMPLETED AFTER 'SYSDATE-1';
EXIT;
EOF
Run this daily to confirm last night's backup completed. If no output, the backup did not run.
4

🔄 Restore a Database from RMAN Backup

Recover a database from a complete failure — step by step

Skill Level
Advanced
Time Required
1–3 hours (depends on DB size)
Category
RMAN Backup & Recovery
Prerequisites
Valid RMAN backup exists · ORACLE_HOME accessible · Sufficient disk for datafiles
Applies To
Oracle 19c / 21c
1

What You Will Learn

Database recovery is the most critical skill an Oracle DBA can have. This guide walks through a complete database restore from RMAN. Read it in advance — in a real recovery scenario, every minute of downtime costs money, so you want to know these steps before you need them.
2

Before You Begin

  • Verify RMAN backups exist and are accessible
    rman target / — then LIST BACKUP SUMMARY; — confirm STATUS=A (Available)
  • Have your ORACLE_HOME, ORACLE_SID, and DB_NAME ready
    If the server crashed, you may need to recreate the Oracle environment
  • This is a DESTRUCTIVE operation — only do this if the database is genuinely failed
    Never run RESTORE/RECOVER on a working database
3

Step-by-Step Instructions

#What to DoCommandExpected Result
1
Assess the situation — what exactly is broken?
 -- If DB won't start, check the error:
sqlplus / as sysdba
STARTUP;
-- Note the ORA- error code before proceeding
-- ORA-01157 = missing datafile
-- ORA-00257 = archiver stuck (disk full)
-- ORA-01578 = block corruption
The error tells you what kind of recovery is needed. Not all failures need a full restore.
2
Start the database in MOUNT state
 STARTUP MOUNT;
-- If even mount fails:
STARTUP NOMOUNT;
-- Then from RMAN:
RESTORE CONTROLFILE FROM AUTOBACKUP;
ALTER DATABASE MOUNT;
MOUNT loads the controlfile but does not open the database. RMAN needs MOUNT state to restore datafiles.
3
Connect to RMAN and list available backups
 rman target /
LIST BACKUP SUMMARY;
LIST BACKUP OF DATABASE;
Confirm you have a usable backup. Note the COMPLETION TIME of the most recent backup — that is your recovery target.
4
Restore the database files from backup
 RESTORE DATABASE;
-- Or restore just a specific tablespace:
RESTORE TABLESPACE users;
-- Or restore a single datafile:
RESTORE DATAFILE '/u01/app/oracle/oradata/ORCL/users01.dbf';
This copies backup files back to their original locations. May take 30 min to 2 hours. Do not interrupt.
5
Recover the database (apply archive logs)
 RECOVER DATABASE;
-- If you get ORA-00283, specify the recovery target:
RECOVER DATABASE UNTIL TIME "TO_DATE('2024-04-04 02:00:00','YYYY-MM-DD HH24:MI:SS')";
RECOVER applies archived redo logs to bring datafiles current. This is what closes the gap between backup time and now.
6
Open the database
 -- If recovering to current time:
ALTER DATABASE OPEN;

-- If you used UNTIL TIME (point-in-time recovery):
ALTER DATABASE OPEN RESETLOGS;
RESETLOGS resets the log sequence — required after any incomplete recovery. The database is now open.
7
Open all PDBs (if CDB)
 ALTER PLUGGABLE DATABASE ALL OPEN;
In a container database, PDBs do not auto-open after recovery. Run this to open them all.
8
Verify database health after recovery
 SELECT NAME, OPEN_MODE, DB_UNIQUE_NAME FROM V\$DATABASE;
SELECT COMP_NAME, STATUS FROM DBA_REGISTRY WHERE STATUS != 'VALID';
SELECT COUNT(*) FROM DBA_OBJECTS WHERE STATUS = 'INVALID';
OPEN_MODE = READ WRITE. Zero invalid components. Investigate any invalid objects with utlrp.sql.
9
Take an immediate backup after recovery
 rman target /
BACKUP DATABASE PLUS ARCHIVELOG DELETE INPUT;
Critical — after RESETLOGS, all previous incremental backups are unusable. Start fresh with a new Level 0.
4

Common Errors & Fixes

Error / SymptomWhat to Do
RMAN-20005: target database name is ambiguousSet the DB_NAME: rman target / dbid=XXXXXXXXX — get DBID from: SELECT DBID FROM V$DATABASE;
ORA-00283: recovery cancelled due to errorsAn archivelog is missing. Check which sequence: LIST ARCHIVELOG ALL; then restore the missing log or use UNTIL SCN.
ORA-01547: warning - RECOVER succeeded but OPEN RESETLOGS would get errorSome datafiles are still inconsistent. Re-run RECOVER DATABASE; until it completes cleanly.
Controlfile is missingIf you have a controlfile autobackup: RESTORE CONTROLFILE FROM AUTOBACKUP; If not, recreate from trace file if available.

🐢 Find Your Slowest SQL Queries

Identify which SQL statements are consuming the most resources right now

Skill Level
Intermediate
Time Required
15 min
Category
Performance Tuning
Prerequisites
DB is OPEN · SELECT on V$SQL and related views · AWR license (Enterprise Edition)
Applies To
Oracle 19c / 21c
1

What You Will Learn

Most Oracle performance problems trace back to a small number of SQL statements consuming disproportionate resources. Finding the top offenders is always the first step in tuning. This guide shows you three methods: real-time top SQL, historical top SQL from AWR, and active session analysis.
2

Before You Begin

  • Know whether you are troubleshooting a current problem or a past event
    Current = use V$SQL. Historical = use AWR (DBA_HIST_SQLSTAT). Identify your target window.
  • Understand the key metrics
    Elapsed time = clock time. CPU time = processor consumed. Buffer gets = logical I/O. Disk reads = physical I/O.
3

Step-by-Step Instructions

#What to DoCommandExpected Result
1
Find current top SQL by elapsed time (real-time)
 SELECT * FROM (
SELECT
sql_id,
ROUND(elapsed_time/1000000,1) elapsed_sec,
ROUND(cpu_time/1000000,1) cpu_sec,
executions,
ROUND(elapsed_time/NULLIF(executions,0)/1000000,3) sec_per_exec,
ROUND(buffer_gets/NULLIF(executions,0)) gets_per_exec,
SUBSTR(sql_text,1,80) sql_preview
FROM v\$sql
WHERE executions > 0
AND elapsed_time > 0
ORDER BY elapsed_time DESC
) WHERE ROWNUM <= 10;
Shows the 10 SQL statements that have consumed the most total time since the instance started. Focus on high sec_per_exec (slow per run) AND high executions (called very often).
2
Find SQL currently running and how long it has been running
 SELECT s.sid, s.serial#, s.username,
s.status, s.wait_class,
s.event,
ROUND(s.last_call_et/60,1) running_minutes,
SUBSTR(q.sql_text,1,100) sql_preview
FROM v\$session s
JOIN v\$sql q ON s.sql_id = q.sql_id
WHERE s.status = 'ACTIVE'
AND s.type = 'USER'
ORDER BY s.last_call_et DESC;
RUNNING_MINUTES tells you how long each active session has been in its current SQL. Anything over a few minutes deserves investigation.
3
Find top SQL by physical reads (disk I/O hogs)
 SELECT * FROM (
SELECT
sql_id,
disk_reads,
executions,
ROUND(disk_reads/NULLIF(executions,0)) reads_per_exec,
buffer_gets,
SUBSTR(sql_text,1,80) sql_preview
FROM v\$sql
WHERE disk_reads > 10000
ORDER BY disk_reads DESC
) WHERE ROWNUM <= 10;
High disk_reads means Oracle is going to disk instead of using buffer cache. Usually indicates a missing index or poor cardinality estimate.
4
Get the full SQL text for a specific sql_id
 SELECT sql_fulltext FROM v\$sql
WHERE sql_id = '&sql_id'
FETCH FIRST 1 ROW ONLY;
V$SQL.SQL_TEXT truncates at 1000 chars. SQL_FULLTEXT contains the complete statement — essential for analysis.
5
Find historical top SQL from AWR (last 24 hours)
 SELECT * FROM (
SELECT
s.sql_id,
ROUND(SUM(s.elapsed_time_delta)/1000000,1) elapsed_sec,
SUM(s.executions_delta) executions,
SUBSTR(t.sql_text,1,80) sql_preview
FROM dba_hist_sqlstat s
JOIN dba_hist_sqltext t ON s.sql_id = t.sql_id
WHERE s.snap_id IN (
SELECT snap_id FROM dba_hist_snapshot
WHERE begin_interval_time > SYSDATE - 1
)
GROUP BY s.sql_id, t.sql_text
ORDER BY elapsed_sec DESC
) WHERE ROWNUM <= 10;
AWR retains historical data. Use this when investigating a problem that occurred earlier — not just the current moment.
6
Find what a specific session is waiting for
 SELECT sid, event, wait_class,
seconds_in_wait, state
FROM v\$session
WHERE sid = &sid;
The EVENT column tells you exactly what the session is waiting for. "db file sequential read" = single-block read (index). "db file scattered read" = multi-block (full table scan).
4

Common Errors & Fixes

Error / SymptomWhat to Do
V$SQL is empty or shows very littleThe SQL area may have been flushed. Use AWR (DBA_HIST_SQLSTAT) for historical data instead.
ORA-01031: insufficient privileges on DBA_HIST viewsNeed SELECT_CATALOG_ROLE or explicit grant: GRANT SELECT ON DBA_HIST_SQLSTAT TO myuser;
Many queries show the same sql_id but different performanceLikely a bind variable issue — different values are getting different plans. Check: SELECT * FROM V$SQL_PLAN WHERE sql_id='...';
Cannot find the slow query despite user complaintsThe session may be idle (waiting for application) not active. Check LAST_CALL_ET and STATUS=INACTIVE — the problem may be in application connection pooling.

 -- KEY SECTIONS TO READ:
-- 1. DB Time & Wait Classes (summary box at top)
-- 2. Top 5 Timed Events (the core of tuning)
-- 3. SQL Ordered by Elapsed Time (find the worst SQL)
-- 4. SQL Ordered by Buffer Gets (find I/O-heavy SQL)
-- 5. Instance Activity Stats (logical/physical reads)
-- 6. Memory Statistics (SGA/PGA sizing)

🗺️ Read an Execution Plan

Understand what Oracle is doing with your SQL — and whether it is efficient

Skill Level
Intermediate
Time Required
20 min
Category
Performance Tuning
Prerequisites
DB is OPEN · SQL query to analyze · DBMS_XPLAN.DISPLAY privilege
Applies To
Oracle 19c / 21c
1

What You Will Learn

An execution plan shows how Oracle will retrieve data for a SQL statement — what indexes it will use, how it will join tables, and how many rows it expects at each step. Reading plans is the core skill of SQL tuning. A bad plan is almost always the cause of slow queries.
2

Before You Begin

  • Have a specific slow SQL query to analyze
    Ideally with the actual sql_id from V$SQL (Guide 09)
  • Understand the difference between EXPLAIN PLAN and actual plan
    EXPLAIN PLAN estimates. DBMS_XPLAN from V$SQL_PLAN shows the ACTUAL plan used at runtime — always prefer the actual plan.
3

Step-by-Step Instructions

#What to DoCommandExpected Result
1
Get the execution plan for a specific sql_id from cache
 SELECT * FROM TABLE(
DBMS_XPLAN.DISPLAY_CURSOR('&sql_id', NULL, 'ALLSTATS LAST')
);
This shows the plan Oracle ACTUALLY used, with real row counts (A-Rows) vs estimated (E-Rows). This is the most reliable method.
2
Generate an estimated EXPLAIN PLAN for a new query
 EXPLAIN PLAN FOR
SELECT e.last_name, d.department_name
FROM employees e
JOIN departments d ON e.department_id = d.department_id
WHERE e.salary > 10000;

-- Then display it:
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY(NULL, NULL, 'TYPICAL'));
Use EXPLAIN PLAN when testing a query before running it. Note: it uses table statistics, not actual data.
3
Read the plan — start from the innermost indented rows
 -- Example plan output:
-- Id | Operation | Name | Rows | Cost
-- 0 | SELECT STATEMENT | | | 45
-- 1 | NESTED LOOPS | | 100 | 45
-- 2 | TABLE ACCESS FULL | EMPLOYEES | 1000 | 40 <-- start here
-- 3 | INDEX RANGE SCAN | DEPT_IDX | 1 | 1
Read from the most-indented row UP. The deepest rows execute first. TABLE ACCESS FULL on a large table = likely missing index.
4
Identify the key operations and what they mean
 -- INDEX UNIQUE SCAN = single row lookup via unique index (excellent)
-- INDEX RANGE SCAN = range of rows via index (good for selective)
-- TABLE ACCESS BY ROWID = row lookup after index (normal)
-- TABLE ACCESS FULL = reading entire table (OK for small, bad for large)
-- NESTED LOOPS = good for small result sets
-- HASH JOIN = good for large joins
-- SORT MERGE JOIN = good when both sides are sorted
-- MERGE JOIN CARTESIAN = almost always a problem (missing join condition)
TABLE ACCESS FULL on tables over 10,000 rows with a WHERE clause usually means a missing index.
5
Compare estimated vs actual rows (find optimizer mistakes)
 -- Look for the Rows(E) vs Rows(A) columns:
-- If E-Rows=1 but A-Rows=100000 the optimizer is badly wrong
-- This causes wrong join methods, wrong indexes, poor performance

-- Fix: gather fresh statistics:
EXEC DBMS_STATS.GATHER_TABLE_STATS('HR','EMPLOYEES',CASCADE=>TRUE);
Big difference between E-Rows and A-Rows = stale statistics. Gather stats and re-check the plan.
6
Check if indexes are being used
 SELECT INDEX_NAME, TABLE_NAME, STATUS,
BLEVEL, LEAF_BLOCKS, CLUSTERING_FACTOR
FROM DBA_INDEXES
WHERE TABLE_NAME = 'EMPLOYEES'
AND STATUS = 'VALID';
If a useful index exists but is not in the plan, Oracle may be ignoring it due to bad stats, column type mismatch, or function on the column (like WHERE UPPER(name)='SMITH').
7
Force a plan to use a specific index (test only)
 SELECT /*+ INDEX(e EMP_SALARY_IDX) */
e.last_name, e.salary
FROM employees e
WHERE e.salary > 10000;
Use hints to test if an index improves performance. NEVER use hints in production code long-term — fix the statistics instead.
4

Common Errors & Fixes

Error / SymptomWhat to Do
Plan shows TABLE ACCESS FULL on a large tableCheck for an index on the WHERE clause columns: SELECT INDEX_NAME FROM DBA_INDEXES WHERE TABLE_NAME='...'; If no index, create one.
E-Rows vs A-Rows differ by orders of magnitudeStale or missing statistics. Run: EXEC DBMS_STATS.GATHER_TABLE_STATS('SCHEMA','TABLE',CASCADE=>TRUE);
Index exists but plan does not use it1. Function on indexed column (WHERE TO_CHAR(date_col,...)). 2. Leading column of composite index not in WHERE. 3. Stats show low selectivity. Use EXPLAIN PLAN to test with a hint.
MERGE JOIN CARTESIAN in the planAlmost always a missing join condition between two tables. Check your WHERE clause for all necessary joins.

What You Will Learn

Data Pump (expdp) is Oracle's high-speed export utility. It creates .dmp files that can be imported into another Oracle database. Unlike the old exp tool, it runs as a server-side job and is 10x-30x faster. Every DBA needs to know this for migrations, clones, and schema backups.
2

  • Create a directory object pointing to the export destination
    mkdir -p /backup/dp && chown oracle:oinstall /backup/dp Then in SQL: CREATE OR REPLACE DIRECTORY dp_dir AS '/backup/dp';
  • Grant privileges to the exporting user
    GRANT READ, WRITE ON DIRECTORY dp_dir TO hr; GRANT DATAPUMP_EXP_FULL_DATABASE TO hr; -- if exporting other schemas
  • Verify sufficient disk space
    df -h /backup/dp — you need at least 1x the schema size
3

#What to DoCommandExpected Result
1
Check schema size before exporting
 SELECT owner, ROUND(SUM(bytes)/1024/1024,1) mb
FROM dba_segments
WHERE owner = 'HR'
GROUP BY owner;
Know how large the export will be so you can verify the output file size afterward.
2
Export a complete schema
 expdp hr/password@ORCL \
SCHEMAS=HR \
DIRECTORY=dp_dir \
DUMPFILE=hr_export_%U.dmp \
LOGFILE=hr_export.log \
COMPRESSION=ALL \
PARALLEL=2
The %U creates multiple dump files for parallel I/O. COMPRESSION=ALL reduces file size significantly. The job runs on the server.
3
Monitor the export job progress
 -- While the job is running, check progress:
SELECT job_name, state, degree,

#What to DoCommandExpected Result

REMAP_SCHEMA maps HR → HR_DEV. REMAP_TABLESPACE maps tablespace names. Use this to import prod data into a dev schema.

📂 Load a Flat File with SQL*Loader

Load CSV and delimited text files directly into Oracle tables

Skill Level
Intermediate
Time Required
20 min (setup) + load time
Category
Data Migration
Prerequisites
DB is OPEN · Table exists in target schema · CSV file accessible on DB server
Applies To
Oracle 19c / 21c
1

What You Will Learn

SQL*Loader is Oracle's utility for loading structured flat files (CSV, TSV, fixed-width) into database tables. It is much faster than INSERT statements and handles huge files efficiently. This is a common DBA task when data arrives from external systems or partner organizations.
2

Before You Begin

  • Have the CSV file on the database server
    Copy via scp or mount a shared filesystem
  • Know the file structure — columns, delimiter, date format
    Run: head -5 yourfile.csv to see the format
  • The target table must already exist
    SQL*Loader does not create tables — create it first with the right column definitions
3

Step-by-Step Instructions

#What to DoCommandExpected Result
1
Inspect the source file
 head -10 /data/employees.csv
# Example output:
# 100,Steven,King,SKING,515.123.4567,17-JUN-2003,AD_PRES,24000
# 101,Neena,Kochhar,NKOCHHAR,515.123.4568,21-SEP-2005,AD_VP,17000
Note: delimiter (comma), any header row, date formats, and whether strings are quoted. These all go in your control file.
2
Create the SQL*Loader control file
 cat > /data/load_employees.ctl << 'EOF'
LOAD DATA
INFILE '/data/employees.csv'
INTO TABLE hr.employees_stg
APPEND
FIELDS TERMINATED BY ','
OPTIONALLY ENCLOSED BY '"'
TRAILING NULLCOLS
(
employee_id,
first_name,
last_name,
email,
phone_number,
hire_date DATE 'DD-MON-YYYY',
job_id,
salary
)
EOF
The control file is the most important part. It maps file columns to table columns. APPEND adds to existing rows. Use TRUNCATE or REPLACE to overwrite.
3
Create a staging table to load into
 CREATE TABLE hr.employees_stg AS
SELECT * FROM hr.employees WHERE 1=0;
-- (creates empty table with same structure)

-- Or create manually:
CREATE TABLE hr.employees_stg (
employee_id NUMBER,
first_name VARCHAR2(20),
last_name VARCHAR2(25),
email VARCHAR2(25),
phone_number VARCHAR2(20),
hire_date DATE,
job_id VARCHAR2(10),
salary NUMBER(8,2)
);
Always load into a staging table first, validate data, then merge/insert into the real table. Never load directly to production tables.
4
Run SQL*Loader
 sqlldr userid=hr/password@ORCL \
control=/data/load_employees.ctl \
log=/data/load_employees.log \
bad=/data/load_employees.bad \
discard=/data/load_employees.dsc \
DIRECT=TRUE \
PARALLEL=TRUE
DIRECT=TRUE bypasses the SQL layer — much faster for large files. BAD file captures rows that failed to load. DISCARD file captures rows that met a WHEN clause condition.
5
Check the load results
 -- Check the log file:
cat /data/load_employees.log | grep -E 'Rows|errors|rejected|loaded'

-- Check rows loaded:
SELECT COUNT(*) FROM hr.employees_stg;

-- Check bad file for rejected rows:
cat /data/load_employees.bad
The log shows: rows loaded, rows rejected (bad data), rows discarded. Check .bad file for any rejected rows — these have error details.
6
Validate data in staging before loading to production
 -- Check for nulls in key columns:
SELECT COUNT(*) FROM hr.employees_stg
WHERE employee_id IS NULL;

-- Check for duplicates:
SELECT employee_id, COUNT(*)
FROM hr.employees_stg
GROUP BY employee_id HAVING COUNT(*) > 1;

-- Compare counts:
SELECT COUNT(*) FROM hr.employees_stg;
Validate row count matches the source file. Check for duplicates and nulls before merging to the real table.
7
Merge from staging to production table
 MERGE INTO hr.employees e
USING hr.employees_stg s
ON (e.employee_id = s.employee_id)
WHEN MATCHED THEN
UPDATE SET e.salary = s.salary, e.job_id = s.job_id
WHEN NOT MATCHED THEN
INSERT (employee_id, first_name, last_name, email, hire_date, job_id, salary)
VALUES (s.employee_id, s.first_name, s.last_name, s.email, s.hire_date, s.job_id, s.salary);
COMMIT;
MERGE handles both inserts and updates in one statement. Always COMMIT after the merge. Verify count afterward.
4

Common Errors & Fixes

Error / SymptomWhat to Do
SQL*Loader-350: Syntax error at line XControl file syntax error. Check: delimiter format, date mask, column names match table, parentheses are balanced.
Rows rejected and written to .bad fileOpen the .bad file + the log file. Common causes: wrong date format, column too long, wrong delimiter, null in NOT NULL column.
DIRECT=TRUE fails with index errorsDirect path load bypasses triggers and some constraints. Use DIRECT=FALSE if the table has complex triggers, or disable/rebuild indexes after load.
Load is very slowTry DIRECT=TRUE if not already set. Check if LOGGING is on — use NOLOGGING on staging table: ALTER TABLE employees_stg NOLOGGING;

1

What You Will Learn

Oracle releases quarterly security and bug-fix patches called Release Updates (RUs). Applying them is a core DBA responsibility. This guide walks you through the complete process: downloading the patch, running pre-checks, applying the binary patch, running datapatch, and validating the result.
2

Before You Begin

  • You have already run Guide 15 and know your current patch level
    You need to know what RU you are on to apply the next one
  • A full RMAN backup has been taken in the last 24 hours
    RMAN> BACKUP DATABASE PLUS ARCHIVELOG — do this FIRST before anything else
  • A maintenance window is scheduled and users have been notified
    This requires database downtime for a standalone (non-RAC) database
3

Step-by-Step Instructions

#What to DoCommandExpected Result
1
Download the patch from My Oracle Support
 -- 1. Log in to https://support.oracle.com
-- 2. Click 'Patches & Updates'
-- 3. Search for: Oracle Database 19c RU
-- or search the specific patch number
-- 4. Download: pXXXXXXX_190000_Linux-x86-64.zip
-- 5. Also download latest OPatch if needed:
-- Patch 6880880 = latest OPatch
Note the patch README.txt — read it before applying. It lists prerequisites, conflicts, and known issues.
2
Upload patch to DB server and unzip
 -- Copy to server:
scp p35943157_190000_Linux-x86-64.zip oracle@dbserver:/patches/

-- On server as oracle user:
cd /patches
unzip p35943157_190000_Linux-x86-64.zip
ls -la
You should see a directory named with the patch number (e.g., 35943157/).
3
Update OPatch if required (check README for minimum version)
 -- Backup existing OPatch first:
cp -r $ORACLE_HOME/OPatch $ORACLE_HOME/OPatch_bak

-- Unzip new OPatch into ORACLE_HOME:
cd $ORACLE_HOME
unzip /patches/p6880880_190000_OPatch.zip

-- Verify new version:
$ORACLE_HOME/OPatch/opatch version
README will say something like "OPatch 12.2.0.1.35 or later required". Confirm new version meets this.
4
Run conflict and space pre-checks
 cd /patches/35943157
$ORACLE_HOME/OPatch/opatch prereq CheckConflictAgainstOHWithDetail -ph ./
$ORACLE_HOME/OPatch/opatch prereq CheckSystemSpace -ph ./
Both checks must pass. If conflict check fails, you may need to rollback a conflicting patch first. Read the README for guidance.
5
Take RMAN backup now (if not done recently)
 rman target /
BACKUP DATABASE PLUS ARCHIVELOG DELETE INPUT;
EXIT;
This is your safety net. Do not skip it. The backup must complete before proceeding.
6
Shut down the database
 sqlplus / as sysdba
SHUTDOWN IMMEDIATE;
EXIT;
lsnrctl stop
Confirm database is down: ps -ef | grep pmon — should show nothing for your SID.
7
Apply the OPatch binary patch
 cd /patches/35943157
$ORACLE_HOME/OPatch/opatch apply -oh $ORACLE_HOME ./
# Takes 15-45 minutes
# Final line must say: OPatch succeeded.
Do NOT interrupt this. If it fails midway: opatch rollback -id PATCHID then investigate the error.
8
Verify patch applied to ORACLE_HOME
 $ORACLE_HOME/OPatch/opatch lspatches | head -3
The new patch ID should appear at the top of the list.

Start the database
 sqlplus / as sysdba
STARTUP;

EXIT;
DB must be OPEN before running datapatch. All PDBs must be open too.
10
Run datapatch to apply SQL changes to DB catalog
 cd $ORACLE_HOME/OPatch
./datapatch -verbose
# Takes 10-30 minutes
# Final line: Datapatch execution has completed.
datapatch patches the database catalog, PL/SQL packages, and dictionary. It is mandatory after every binary patch.
11
Recompile any invalid objects
 sqlplus / as sysdba
@$ORACLE_HOME/rdbms/admin/utlrp.sql
EXIT;
Runs in the background. Typically completes in a few minutes. Repeat until: SELECT COUNT(*) FROM DBA_OBJECTS WHERE STATUS='INVALID'; returns 0.
12
Validate the patch in the database
 SELECT patch_id, action, status, description
FROM dba_registry_sqlpatch
WHERE action='APPLY'
ORDER BY action_time DESC
FETCH FIRST 3 ROWS ONLY;

SELECT comp_name, version, status
FROM dba_registry
WHERE status != 'VALID';
datapatch rows must show STATUS=SUCCESS. dba_registry query must return zero rows.
13
Restart listener and test connectivity

tnsping ORCL
sqlplus app_user/password@ORCL
Confirm applications can connect. Monitor alert log for 15 minutes after opening to applications: tail -f alert_ORCL.log
4

#What to DoCommandExpected Result

 id oracle
# Expected output:
# uid=54321(oracle) gid=54321(oinstall) groups=54321(oinstall),54322(dba)

2
Check what Oracle instances are registered
 cat /etc/oratab
Format: SID:ORACLE_HOME:Y/N. The Y/N controls whether dbstart/dbshut manages this instance. Change N to Y for auto-startup.
3
Set up the oracle user bash_profile
 su - oracle
cat ~/.bash_profile

export LD_LIBRARY_PATH=$ORACLE_HOME/lib

If these variables are missing, add them. NLS_DATE_FORMAT makes SQL date output human-readable.

 source ~/.bash_profile
echo $ORACLE_HOME
echo $ORACLE_SID
which sqlplus

These aliases save significant typing. alias + alert lets you tail the alert log with a single word.

# kernel.shmmax >= RAM/2 (in bytes)
# kernel.shmall >= shmmax/4096
# fs.file-max >= 6815744
# kernel.sem: 250 32000 100 128

# open files: 65536
# max user processes: 16384

1

What You Will Learn

Oracle generates a lot of files — trace files, core dumps, alert logs, archivelogs, audit files. Left unmanaged, these fill up filesystems and cause database outages (a full archivelog destination stops the database cold). This guide shows how to find space hogs, clean them up safely, and set up automation.
2

Before You Begin

  • Know your ORACLE_BASE and archivelog destination
    SELECT VALUE FROM V\$PARAMETER WHERE NAME IN ('db_recovery_file_dest','log_archive_dest_1');
  • Check current disk usage first
    df -h — note which filesystems are nearly full
3

Step-by-Step Instructions

#What to DoCommandExpected Result
1
Find the biggest space consumers
 du -sh $ORACLE_BASE/diag/rdbms/*/*/trace/
du -sh $ORACLE_BASE/diag/rdbms/*/*/incident/
du -sh /arch/ 2>/dev/null || du -sh $ORACLE_BASE/fast_recovery_area/
du -sh $ORACLE_BASE/admin/*/adump/
These four directories are the usual suspects. Trace files, incident dumps, archivelogs, and audit files.
2
Check the Fast Recovery Area usage from SQL
 SELECT
space_limit/1024/1024/1024 limit_gb,
space_used/1024/1024/1024 used_gb,
ROUND(space_used/space_limit*100,1) pct_used
FROM v\$recovery_file_dest;

-- See breakdown by file type:
SELECT file_type, percent_space_used, number_of_files
FROM v\$recovery_area_usage
ORDER BY percent_space_used DESC;
If FRA is over 85% full, Oracle will stop writing archivelogs and the database will hang. Act immediately.
3
Clean up obsolete RMAN backups (safest method)
 rman target /
DELETE NOPROMPT OBSOLETE;
DELETE NOPROMPT ARCHIVELOG ALL COMPLETED BEFORE 'SYSDATE-7';
EXIT;
RMAN knows which files are needed for recovery. Let it delete — never manually rm RMAN backup files. DELETE OBSOLETE removes backups outside the retention policy.
4
Clean up trace files using adrci
 adrci
adrci> show homes
adrci> set homepath diag/rdbms/orcl/ORCL
adrci> purge -age 20160 -type TRACE
adrci> purge -age 20160 -type ALERT
adrci> quit
# 20160 minutes = 14 days
adrci purge is the Oracle-approved way to clean diagnostic files. It respects incident files that should be kept.
5
Find and review large incident trace files before deleting
 find $ORACLE_BASE/diag -name '*.trc' -size +50M -exec ls -lh {} \;
find $ORACLE_BASE/diag/rdbms -name '*.trm' -mtime +30 -ls | head -20
Large .trc files are usually from ORA-600 or ORA-7445 internal errors. Check them before deleting — they may contain important diagnostic info.
6
Clean up audit files (adump directory)
 find $ORACLE_BASE/admin/ORCL/adump/ -name '*.aud' -mtime +30 -exec rm {} \;
# Or:
find $ORACLE_BASE/admin/ORCL/adump/ -name '*.aud' -mtime +30 -delete
ls -lh $ORACLE_BASE/admin/ORCL/adump/ | wc -l
Audit files accumulate quickly on busy databases. Files older than 30 days are generally safe to delete after confirming with your security team.
7
Set up automatic cleanup with cron
 cat > /scripts/oracle_cleanup.sh << 'EOF'
#!/bin/bash
export ORACLE_HOME=/u01/app/oracle/product/19c/db
export ORACLE_SID=ORCL
export PATH=$ORACLE_HOME/bin:$PATH

# Clean RMAN obsolete backups
rman target / << RMAN
DELETE NOPROMPT OBSOLETE;
DELETE NOPROMPT ARCHIVELOG ALL COMPLETED BEFORE 'SYSDATE-7';
EXIT;
RMAN

# Clean trace files > 14 days
adrci << ADRCI
set homepath diag/rdbms/orcl/ORCL
purge -age 20160 -type TRACE
ADRCI
EOF
chmod +x /scripts/oracle_cleanup.sh

# Schedule: every Sunday at 3 AM
crontab -e
# 0 3 * * 0 /scripts/oracle_cleanup.sh
Automate weekly cleanup. Always log the output: add >> /logs/cleanup_$(date +%Y%m%d).log 2>&1 to the cron line.
8
Verify disk space improved
 df -h
du -sh $ORACLE_BASE/diag/rdbms/*/*/trace/

-- Confirm FRA usage dropped:
SELECT ROUND(space_used/space_limit*100,1) pct_used
FROM v\$recovery_file_dest;
FRA should be below 75%. OS filesystem should have comfortable headroom. Set up an alert if it exceeds 80%.
4

Common Errors & Fixes

Error / SymptomWhat to Do
Database hung with "archiver stuck" messageArchivelog destination is 100% full. IMMEDIATE: Delete old archivelogs via RMAN or free disk space. The database will resume automatically once space is available.
Cannot delete files — permission deniedFiles must be deleted as oracle OS user (not root for Oracle files). Check: ls -la on the directory.

What You Will Learn

Manual checks do not scale. A good DBA automates routine health checks so problems are caught overnight before users arrive. This guide builds a complete daily health check script covering instance status, tablespace usage, failed jobs, and recent ORA- errors — and optionally emails you the results.

  • Environment variables set correctly in bash_profile

  • Know how to run sqlplus from the shell command line
    sqlplus / as sysdba — should work without prompts when run as oracle user
  • Decide where logs will be stored

3

Step-by-Step Instructions

#What to DoCommandExpected Result
1
Understand the pattern: sqlplus heredoc in bash
 #!/bin/bash
# The << EOF...EOF pattern sends SQL commands to sqlplus:
sqlplus -s / as sysdba << EOF
SET HEADING OFF FEEDBACK OFF PAGESIZE 0
SELECT 'DB Status: ' || STATUS FROM V\$INSTANCE;
EXIT;
EOF
The -s flag runs sqlplus silently (no banner). SET HEADING OFF removes column headers. The \ before $ prevents bash from interpreting it.
2
Build the health check script step by step
 cat > /scripts/db_healthcheck.sh << 'SCRIPT'
#!/bin/bash
# ===========================================
# Oracle Daily Health Check
# Run as: oracle OS user via cron
# ===========================================
export ORACLE_HOME=/u01/app/oracle/product/19c/db
export ORACLE_SID=ORCL
export PATH=$ORACLE_HOME/bin:$PATH

DATE=$(date '+%Y-%m-%d %H:%M')
LOGFILE=/logs/oracle_health/healthcheck_$(date +%Y%m%d).log

echo "=== Oracle Health Check: $DATE ===" > $LOGFILE
echo "" >> $LOGFILE
SCRIPT
Always include the Oracle environment variables at the top of every script. Never assume they are set.
3
Add instance status check
 cat >> /scripts/db_healthcheck.sh << 'SCRIPT'
echo "--- INSTANCE STATUS ---" >> $LOGFILE
sqlplus -s / as sysdba >> $LOGFILE 2>&1 << SQL
SET LINESIZE 100 PAGESIZE 20
SELECT INSTANCE_NAME, HOST_NAME, VERSION_FULL, STATUS,
TO_CHAR(STARTUP_TIME,'DD-MON-YYYY HH24:MI') started
FROM V\$INSTANCE;
EXIT;
SQL
echo "" >> $LOGFILE
SCRIPT
This checks the instance is up and shows version and startup time. If the DB is down, sqlplus will output an error to the log.
4
Add tablespace usage check
 cat >> /scripts/db_healthcheck.sh << 'SCRIPT'
echo "--- TABLESPACE USAGE (>70% full) ---" >> $LOGFILE
sqlplus -s / as sysdba >> $LOGFILE 2>&1 << SQL
SET LINESIZE 120 PAGESIZE 30 COLSEP '|'
COLUMN tablespace_name FORMAT A20
SELECT t.tablespace_name,
ROUND(SUM(d.bytes)/1024/1024,0) total_mb,
ROUND(SUM(d.bytes - NVL(f.bytes,0))/1024/1024,0) used_mb,
ROUND(NVL(SUM(f.bytes),0)/SUM(d.bytes)*100,1) pct_free
FROM dba_tablespaces t
JOIN dba_data_files d ON t.tablespace_name=d.tablespace_name
LEFT JOIN (SELECT tablespace_name, SUM(bytes) bytes FROM dba_free_space GROUP BY tablespace_name) f
ON t.tablespace_name=f.tablespace_name
GROUP BY t.tablespace_name
HAVING ROUND(NVL(SUM(f.bytes),0)/SUM(d.bytes)*100,1) < 30
ORDER BY pct_free;
EXIT;
SQL
echo "" >> $LOGFILE
SCRIPT
HAVING < 30 means: only show tablespaces with less than 30% free (i.e., over 70% full). Blank output = all tablespaces healthy.
5
Add recent ORA- error check from alert log
 cat >> /scripts/db_healthcheck.sh << 'SCRIPT'
echo "--- ORA- ERRORS (last 24 hours) ---" >> $LOGFILE

grep 'ORA-' $ALERT_LOG | \
grep "$(date +%Y-%m-%d)\|$(date -d 'yesterday' +%Y-%m-%d)" | \
grep -oP 'ORA-[0-9]+' | sort | uniq -c | sort -rn \
>> $LOGFILE 2>&1
echo "" >> $LOGFILE
SCRIPT
Summarizes unique ORA- errors from the last 24 hours with counts. No output = clean alert log.
6
Add failed scheduler jobs check
 cat >> /scripts/db_healthcheck.sh << 'SCRIPT'
echo "--- FAILED SCHEDULER JOBS ---" >> $LOGFILE
sqlplus -s / as sysdba >> $LOGFILE 2>&1 << SQL
SET LINESIZE 100 PAGESIZE 20
SELECT JOB_NAME, LOG_DATE, STATUS, ERROR#
FROM DBA_SCHEDULER_JOB_RUN_DETAILS
WHERE STATUS != 'SUCCEEDED'
AND LOG_DATE > SYSDATE - 1
ORDER BY LOG_DATE DESC;
EXIT;
SQL
echo "" >> $LOGFILE
SCRIPT
Catches any overnight scheduled jobs that failed — common source of silent data issues.
7
Make executable and test
 chmod +x /scripts/db_healthcheck.sh
/scripts/db_healthcheck.sh
cat /logs/oracle_health/healthcheck_$(date +%Y%m%d).log
Run it manually first. The log file should have all four sections with data. Fix any errors before scheduling.
8
Schedule with cron and optionally email results
 crontab -e
# Run at 6 AM every day:
0 6 * * * /scripts/db_healthcheck.sh

# To email results (if mail is configured):
# 0 6 * * * /scripts/db_healthcheck.sh && mail -s 'Oracle Health Check' youremail@company.com < /logs/oracle_health/healthcheck_$(date +\%Y\%m\%d).log
The DBA team gets the report in their inbox every morning before the business day starts.
4

Error / SymptomWhat to Do

  • Get the storage requirement from the application team
    How much data will the app store? Start with that + 20% buffer for indexes and growth.

-- !df -h /u01

Create a dedicated tablespace for the application
CREATE TABLESPACE fusion_ts
DATAFILE '+DATA01' SIZE 200M
REUSE AUTOEXTEND ON NEXT 100M MAXSIZE 3G;

3

CREATE PROFILE app_profile LIMIT

PASSWORD_LIFE_TIME 180
IDLE_TIME 30

-- Check first:
SELECT USERNAME, ACCOUNT_STATUS FROM DBA_USERS
WHERE USERNAME = 'FUSION';

DROP USER fusion CASCADE;
If user does not exist, ORA-01918 appears — that is expected, continue. CASCADE removes all objects owned by the user.
6

CREATE USER fusion
IDENTIFIED BY "SecurePass_123!"

TEMPORARY TABLESPACE temp
PROFILE app_profile
ACCOUNT UNLOCK;
User created. The account is open and associated with the dedicated tablespace and security profile.
7
Grant quota on the tablespace
ALTER USER fusion QUOTA UNLIMITED ON fusion_ts;
User altered. Without a quota, the user cannot create objects even though the tablespace exists.
8
Create a role and grant privileges to it
CREATE ROLE fusion_role;

GRANT CREATE VIEW, CREATE SYNONYM TO fusion_role;
Role created. Using a role makes it easy to manage and audit privileges — grant/revoke the role instead of individual privileges.

GRANT fusion_role TO fusion;
ALTER USER fusion DEFAULT ROLE ALL;
Grant succeeded. The user now has all privileges through the role.

WHERE USERNAME = 'FUSION';

CONNECT fusion/"SecurePass_123!"

USER is "FUSION". DEFAULT_TABLESPACE = FUSION_TS. Connection works — account is ready.

-- Username: FUSION

Send credentials via your organization's secure channel. Document the account creation in your change management system.

DB is OPEN · SYSDBA or user with GRANT ANY PRIVILEGE · Target schema exists
Applies To
Oracle 19c / 21c
1

What You Will Learn

Granting privileges is how you control database security. There are two approaches: granting directly to a user (quick but hard to manage at scale), or granting to a role then assigning the role to users (the correct approach for any production environment). This guide covers both, with real examples from a DBA training environment.
2

Before You Begin

  • Know what the user needs to do
    SELECT? INSERT? Run procedures? Each task maps to a specific privilege.
  • Know which schema owns the objects
    The granting user must own the objects or have WITH GRANT OPTION
  • Understand the principle of least privilege
    Only grant what is needed. Never grant DBA unless absolutely necessary.
3

Step-by-Step Instructions

#What to DoCommandExpected Result
1
View all system privileges available
SELECT DISTINCT PRIVILEGE FROM DBA_SYS_PRIVS
ORDER BY PRIVILEGE;
Lists all system-level privileges. These control actions like CREATE TABLE, CREATE SESSION, CREATE USER.
2
View object privileges on a specific table
SELECT GRANTEE, PRIVILEGE, GRANTABLE
FROM DBA_TAB_PRIVS
WHERE TABLE_NAME = 'EMPLOYEES'
AND OWNER = 'HR';
Shows who has access to the HR.EMPLOYEES table and what they can do.
3
View all objects in a schema
SELECT OBJECT_TYPE, OBJECT_NAME, STATUS
FROM DBA_OBJECTS
WHERE OWNER = 'APP_USER'
ORDER BY OBJECT_TYPE, OBJECT_NAME;
Lists every object (table, view, procedure, index) owned by APP_USER.
4
Grant direct object privileges to a user
GRANT SELECT, INSERT, UPDATE, DELETE
ON app_user.customer TO app_mgnr;
GRANT SELECT, INSERT, UPDATE, DELETE
ON app_user.orders TO app_mgnr;
GRANT SELECT, INSERT, UPDATE, DELETE
ON app_user.product TO app_mgnr;
Grant succeeded for each. Direct grants work but are hard to manage when users change roles.
5
Verify the user can access the data
CONNECT app_mgnr/password
SELECT COUNT(*) FROM app_user.customer;
SELECT COUNT(*) FROM app_user.product;
Returns row counts. If you see ORA-00942 (table not found), the grant did not work — check the schema.owner.table syntax.
6
Better approach: create a role
CONNECT / as sysdba
CREATE ROLE appuser_dev;
Role created. Now assign privileges to the role, not directly to users.
7
Grant object privileges to the role
GRANT SELECT, INSERT, UPDATE, DELETE
ON app_user.customer TO appuser_dev;
GRANT SELECT, INSERT, UPDATE, DELETE
ON app_user.orders TO appuser_dev;
GRANT SELECT, INSERT, UPDATE, DELETE
ON app_user.product TO appuser_dev;
GRANT SELECT, INSERT, UPDATE, DELETE
ON app_user.order_item TO appuser_dev;
GRANT SELECT, INSERT, UPDATE, DELETE
ON app_user.payment TO appuser_dev;
Each GRANT returns "Grant succeeded". The role now bundles all access in one place.
8
Grant the role to the user
GRANT appuser_dev TO app_mgnr;
ALTER USER app_mgnr DEFAULT ROLE ALL;
Grant succeeded. User altered. The user automatically gets all privileges defined in the role.
9
Verify role assignment
SELECT GRANTED_ROLE FROM DBA_ROLE_PRIVS
WHERE GRANTEE = 'APP_MGNR';
Returns: APPUSER_DEV. Confirms the role is correctly assigned.
10
Test access as the role-based user
CONNECT app_mgnr/password
SELECT COUNT(*) FROM app_user.customer;
DESC app_user.product;
Row count returned. Table structure visible. Role-based access is working correctly.
4

Common Errors & Fixes

Error / SymptomWhat to Do
ORA-00942: table or view does not existThe schema.table reference is wrong, or the granting user does not own the object. Check: SELECT OWNER, TABLE_NAME FROM DBA_TABLES WHERE TABLE_NAME='CUSTOMER';
ORA-01031: insufficient privileges when grantingYou need GRANT ANY OBJECT PRIVILEGE or must own the object. Connect as SYSDBA or the schema owner.
Privilege granted but user still gets ORA-00942User needs to reference the table as schema.table_name (e.g., app_user.customer) unless a synonym exists.
Role not active after grantRun: ALTER USER app_mgnr DEFAULT ROLE ALL; — ensures all granted roles are active on login.

👤 Manage User Profiles

Control password policies and session limits with Oracle profiles

Skill Level
Intermediate
Time Required
20 min
Category
Database Administration
Prerequisites
DB is OPEN · SYSDBA privilege · DBA_PROFILES view access
Applies To
Oracle 19c / 21c
1

What You Will Learn

A profile is a set of limits assigned to a database user that controls password complexity, how long a password lasts, how many failed logins are allowed before lockout, and session resource limits. Every user is assigned the DEFAULT profile unless you specify otherwise. This guide shows how to view, create, and assign profiles.
2

Before You Begin

  • Know the security requirements for the user
    How many failed logins before lockout? How often must password be changed? Session idle timeout?
  • The DEFAULT profile settings on your database
    Run: SELECT RESOURCE_NAME, LIMIT FROM DBA_PROFILES WHERE PROFILE='DEFAULT'; — know what you are changing from
3

Step-by-Step Instructions

#What to DoCommandExpected Result
1
Check what profiles exist in the database
SELECT DISTINCT PROFILE FROM DBA_PROFILES
ORDER BY PROFILE;
Lists all profiles. By default: DEFAULT and ORA_STIG_PROFILE. Any custom profiles you have created also appear here.
2
View the settings for the DEFAULT profile
SELECT RESOURCE_NAME, RESOURCE_TYPE, LIMIT
FROM DBA_PROFILES
WHERE PROFILE = 'DEFAULT'
ORDER BY RESOURCE_TYPE, RESOURCE_NAME;
Shows all 18 resource settings. Key ones: FAILED_LOGIN_ATTEMPTS=10, PASSWORD_LIFE_TIME=180, PASSWORD_LOCK_TIME=1 day.
3
Check a specific user's profile assignment
SELECT USERNAME, ACCOUNT_STATUS, PROFILE, CREATED
FROM DBA_USERS
WHERE USERNAME = 'EMPIRE';
Shows the profile name assigned to the user. DEFAULT means no custom profile has been applied.
4
Create a custom application profile
CREATE PROFILE app_profile LIMIT
SESSIONS_PER_USER 2
IDLE_TIME 30
FAILED_LOGIN_ATTEMPTS 3
PASSWORD_LIFE_TIME 365;
Profile created. This sets: max 2 concurrent sessions, 30-min idle timeout, lock after 3 bad passwords, password expires yearly.
5
Verify the new profile was created
SELECT RESOURCE_NAME, LIMIT FROM DBA_PROFILES
WHERE PROFILE = 'APP_PROFILE'
ORDER BY RESOURCE_TYPE;
Lists the limits you defined. Resources not specified inherit from DEFAULT (shown as "DEFAULT" in the LIMIT column).
6
Assign the profile to a user
ALTER USER empire PROFILE app_profile;
User altered. The new profile takes effect immediately — no restart needed.
7
Verify the profile assignment
SELECT USERNAME, PROFILE FROM DBA_USERS
WHERE USERNAME = 'EMPIRE';
PROFILE column now shows APP_PROFILE instead of DEFAULT.
8
Test the failed login limit
-- Deliberately try wrong passwords:
-- CONNECT empire/wrongpass (1st fail)
-- CONNECT empire/wrongpass (2nd fail)
-- CONNECT empire/wrongpass (3rd fail)
-- CONNECT empire/wrongpass → ORA-28000: account locked
After 3 failures, ORA-28000 appears. This confirms the profile's FAILED_LOGIN_ATTEMPTS=3 is working.
9
Unlock the account after lockout
ALTER USER empire ACCOUNT UNLOCK;
User altered. Account is restored. In production, also check who/what caused the lockout before unlocking.
10
Modify an existing profile
ALTER PROFILE app_profile LIMIT
FAILED_LOGIN_ATTEMPTS 5
PASSWORD_LIFE_TIME 90;
Profile altered. Changes take effect immediately for all users assigned to this profile.
4

Common Errors & Fixes

Error / SymptomWhat to Do
ORA-28000: The account is lockedAccount locked due to failed login attempts. Unlock with: ALTER USER username ACCOUNT UNLOCK; Then investigate what caused the failures.
Profile change has no effect on sessionsProfile limits apply to new sessions. Existing connected sessions are not affected until they reconnect.
ORA-02379: profile already existsProfile name already in use. Either use the existing one (ALTER PROFILE) or choose a different name.
IDLE_TIME not enforcedIDLE_TIME requires RESOURCE_LIMIT=TRUE: ALTER SYSTEM SET RESOURCE_LIMIT=TRUE; Check: SHOW PARAMETER RESOURCE_LIMIT

🔍 Investigate and Resolve Account Lockouts

Find which process is repeatedly locking an application account

Skill Level
Intermediate
Time Required
15 min
Category
Database Administration
Prerequisites
DB is OPEN · SYSDBA · Auditing enabled (default in 19c)
Applies To
Oracle 19c / 21c
1

What You Will Learn

Application users sometimes report they keep getting locked out even after you unlock their account. A simple unlock is not enough — you need to find the process that is trying to authenticate with a bad password. This is usually a misconfigured application, a service account with an old password, or a scheduled job. This guide shows you how to find it.
2

Before You Begin

  • Know the username of the locked account
    Usually reported by the application team or user
  • Auditing must be enabled (default in Oracle 19c)
    SELECT VALUE FROM V$PARAMETER WHERE NAME='audit_trail'; — should not be NONE
  • Know when the lockouts are happening
    Approximate time helps narrow the audit trail search
3

Step-by-Step Instructions

#What to DoCommandExpected Result
1
Set environment and connect as SYSDBA
oraenv # enter your SID
sqlplus / as sysdba
Connected. Confirm which database you are on: SELECT NAME FROM V$DATABASE;
2
Check the database status
SELECT NAME, OPEN_MODE, LOG_MODE FROM V$DATABASE;
Confirm DB is READ WRITE and in ARCHIVELOG mode — the expected state for a production database.
3
Check if the account is currently locked
SELECT USERNAME, ACCOUNT_STATUS, LOCK_DATE
FROM DBA_USERS
WHERE USERNAME = 'APP_USER';
ACCOUNT_STATUS = LOCKED shows when it was locked. LOCK_DATE shows exactly when it happened.
4
Query audit trail for failed login attempts
COL USERNAME FORMAT A25
COL OS_USERNAME FORMAT A25
COL USERHOST FORMAT A25

SELECT USERNAME, OS_USERNAME, USERHOST,
EXTENDED_TIMESTAMP
FROM SYS.DBA_AUDIT_SESSION
WHERE RETURNCODE != 0
AND USERNAME = 'APP_USER'
AND EXTENDED_TIMESTAMP > (SYSTIMESTAMP - 3)
ORDER BY EXTENDED_TIMESTAMP DESC;
Returns rows showing: which OS user, from which host, and exactly when each failed attempt happened. The USERHOST column identifies the server or workstation causing the lockouts.
5
Interpret the results
-- USERHOST = application_server01 → the app server has wrong credentials
-- OS_USERNAME = oracle → a DBA script has hardcoded wrong password
-- USERHOST = workstation-jsmith → a user has wrong saved password
-- No rows selected → either auditing off or no failed attempts in 3 days
Identify the source. Most common cause: application connection pool configured with an old password after a password rotation.
6
Temporarily unlock the account while investigating
ALTER USER app_user ACCOUNT UNLOCK;
User altered. This buys time while you fix the root cause. The account will lock again if the bad attempts continue.
7
Fix the root cause based on findings
-- If app server: update connection string/config with correct password
-- If scheduled job: update the job's stored credentials
-- If user workstation: ask user to clear saved passwords and re-enter
Fix first, THEN unlock permanently. Unlocking without fixing root cause is a temporary solution.
8
Verify no more failed attempts after fix
SELECT COUNT(*) FROM SYS.DBA_AUDIT_SESSION
WHERE RETURNCODE != 0
AND USERNAME = 'APP_USER'
AND EXTENDED_TIMESTAMP > SYSTIMESTAMP - 1/24;
Zero rows = no failed attempts in the last hour. Root cause is resolved.
4

Common Errors & Fixes

Error / SymptomWhat to Do
No rows selected from DBA_AUDIT_SESSIONAuditing may not be capturing session events. Check: SELECT * FROM DBA_AUDIT_TRAIL WHERE ROWNUM<5; Or enable: AUDIT CREATE SESSION WHENEVER NOT SUCCESSFUL;
Account keeps relocking after unlockRoot cause not fixed. Something is still attempting authentication with wrong credentials. Check USERHOST and OS_USERNAME in audit trail.
ORA-00942: table or view does not exist on DBA_AUDIT_SESSIONNeed SELECT_CATALOG_ROLE or explicit grant: GRANT SELECT ON SYS.DBA_AUDIT_SESSION TO your_user;
EXTENDED_TIMESTAMP column not foundUse older column: TIMESTAMP$ — or query DBA_AUDIT_TRAIL instead of DBA_AUDIT_SESSION

📐 Check Tablespace Assignment and Storage for Users

Verify where user data lives and how much space is available

Skill Level
Beginner
Time Required
10 min
Category
Database Administration
Prerequisites
DB is OPEN · SELECT on DBA_USERS, DBA_TABLESPACES, V$ASM_DISKGROUP
Applies To
Oracle 19c / 21c
1

What You Will Learn

Knowing which tablespace a user's data is stored in, and how much space is available, is a daily DBA task. This is especially important before creating new users (to ensure there is space), and when troubleshooting ORA-01536 space quota errors. This guide shows the key queries for tablespace and user storage management.
2

Before You Begin

  • Know the username you are checking
    Case-sensitive in Oracle — always use UPPER case for username lookups
  • Know if your storage is filesystem-based or ASM
    ASM = use V$ASM_DISKGROUP. Filesystem = use df -h from OS
3

Step-by-Step Instructions

#What to DoCommandExpected Result
1
View all tablespaces in the database
SELECT TABLESPACE_NAME, STATUS, CONTENTS
FROM DBA_TABLESPACES
ORDER BY TABLESPACE_NAME;
Lists all tablespaces. STATUS should be ONLINE for all active ones. CONTENTS shows PERMANENT, TEMPORARY, or UNDO.
2
Check default tablespace for a specific user
SELECT USERNAME, DEFAULT_TABLESPACE,
TEMPORARY_TABLESPACE, PROFILE, ACCOUNT_STATUS
FROM DBA_USERS
WHERE USERNAME = 'UNIDB';
DEFAULT_TABLESPACE shows where the user's objects are stored. TEMPORARY_TABLESPACE shows where sort operations happen (usually TEMP).
3
Check available space on filesystem tablespace
SELECT DF.TABLESPACE_NAME,
ROUND(DF.BYTES/1024/1024,0) total_mb,
ROUND(NVL(FS.BYTES,0)/1024/1024,0) free_mb,
ROUND((1 - NVL(FS.BYTES,0)/DF.BYTES)*100,1) pct_used
FROM
(SELECT TABLESPACE_NAME, SUM(BYTES) BYTES FROM DBA_DATA_FILES GROUP BY TABLESPACE_NAME) DF
LEFT JOIN
(SELECT TABLESPACE_NAME, SUM(BYTES) BYTES FROM DBA_FREE_SPACE GROUP BY TABLESPACE_NAME) FS
ON DF.TABLESPACE_NAME = FS.TABLESPACE_NAME
ORDER BY PCT_USED DESC;
Shows every tablespace with total, free, and % used. Any tablespace over 85% full needs attention.
4
Check ASM diskgroup space (if using ASM)
SELECT NAME, TOTAL_MB, FREE_MB,
ROUND(FREE_MB/TOTAL_MB*100,1) pct_free,
STATE
FROM V$ASM_DISKGROUP;
Shows ASM disk groups (DATA01, FRA01, etc.) with total and free MB. This is the underlying storage that tablespace datafiles live on.
5
Check a user's quota on their tablespace
SELECT USERNAME, TABLESPACE_NAME,
BYTES/1024/1024 used_mb,
MAX_BYTES/1024/1024 quota_mb
FROM DBA_TS_QUOTAS
WHERE USERNAME = 'UNIDB';
Shows how much the user has used vs their quota. MAX_BYTES=-1 means UNLIMITED quota.
6
Grant a user quota on a tablespace
ALTER USER unidb QUOTA 500M ON users;
-- Or unlimited:
ALTER USER unidb QUOTA UNLIMITED ON fusion_ts;
User altered. The user can now create objects up to their quota limit. Without quota, they will get ORA-01536 on INSERT/CREATE.
7
Find which tablespace a specific table lives in
SELECT TABLE_NAME, TABLESPACE_NAME,
NUM_ROWS, LAST_ANALYZED
FROM DBA_TABLES
WHERE OWNER = 'APP_USER'
ORDER BY TABLE_NAME;
Shows each table and its tablespace. LAST_ANALYZED shows when statistics were last gathered — important for query performance.
4

Common Errors & Fixes

Error / SymptomWhat to Do
ORA-01536: space quota exceeded on tablespaceUser has hit their quota. Run: ALTER USER username QUOTA UNLIMITED ON tablespace_name;
DBA_TABLESPACES shows STATUS=OFFLINEThe tablespace is offline — objects inside are inaccessible. Bring online: ALTER TABLESPACE ts_name ONLINE; then investigate why it went offline.
V$ASM_DISKGROUP returns no rowsEither not using ASM, or connected to a database without ASM access. Check: SHOW PARAMETER DB_CREATE_FILE_DEST
User's DEFAULT_TABLESPACE is SYSTEMThis is a configuration mistake. Fix it: ALTER USER username DEFAULT TABLESPACE correct_ts; Then move their objects if needed.

🗃️ Create and Manage Tablespaces (with ASM)

Add storage for applications using Oracle ASM diskgroups

Skill Level
Intermediate
Time Required
20 min
Category
Database Administration
Prerequisites
DB is OPEN · SYSDBA · ASM diskgroup available (or filesystem path)
Applies To
Oracle 19c / 21c with ASM
1

What You Will Learn

This guide covers creating and managing tablespaces in an ASM (Automatic Storage Management) environment — the standard in production Oracle databases. With ASM, you specify the diskgroup name (+DATA01) instead of a file path, and Oracle manages the physical file placement automatically. You will also learn how to add datafiles when a tablespace fills up.
2

Before You Begin

  • Know which ASM diskgroup to use
    SELECT NAME, FREE_MB, TOTAL_MB FROM V$ASM_DISKGROUP; — use DATA01 or equivalent for user data
  • Know the sizing requirement
    How large should the initial tablespace be? How large can it grow?
  • Confirm the diskgroup has enough free space
    New tablespace size must fit within the diskgroup's FREE_MB
3

Step-by-Step Instructions

#What to DoCommandExpected Result
1
Connect and check the database environment
. oraenv # enter your SID at prompt
sqlplus / as sysdba
SELECT NAME, OPEN_MODE FROM V$DATABASE;
Connected. Database is READ WRITE — safe to make changes.
2
Check existing datafiles and their tablespaces
SELECT FILE_NAME, TABLESPACE_NAME
FROM DBA_DATA_FILES
ORDER BY TABLESPACE_NAME;
Lists all datafiles with their ASM paths (+DATA01/DBNAME/DATAFILE/...) and which tablespace each belongs to.
3
Check all current tablespaces
SELECT TABLESPACE_NAME FROM DBA_TABLESPACES;
Lists all tablespaces. Confirm your new tablespace name does not already exist.
4
Create a new tablespace (ASM)
CREATE TABLESPACE fusion
DATAFILE '+DATA01' SIZE 100M;
Tablespace created. Oracle assigns an auto-generated ASM filename. Verify: SELECT TABLESPACE_NAME FROM DBA_TABLESPACES;
5
Verify the datafile that was created
SELECT FILE_NAME, TABLESPACE_NAME
FROM DBA_DATA_FILES
WHERE TABLESPACE_NAME = 'FUSION';
Shows the full ASM path like: +DATA01/DBNAME/DATAFILE/fusion.344.1164235091. Every datafile in ASM gets a unique numeric suffix.
6
Add a second datafile to an existing tablespace
ALTER TABLESPACE fusion
ADD DATAFILE '+DATA01' SIZE 100M;
Tablespace altered. The new file gets a different ASM name even though they are in the same tablespace — each file is unique.
7
Add multiple datafiles at once (shortcut)
ALTER TABLESPACE fusion ADD DATAFILE '+DATA01' SIZE 100M;
ALTER TABLESPACE fusion ADD DATAFILE '+DATA01' SIZE 100M;
ALTER TABLESPACE fusion ADD DATAFILE '+DATA01' SIZE 100M;
Run each ALTER TABLESPACE separately. Tablespace altered for each. You can verify: SELECT COUNT(*) FROM DBA_DATA_FILES WHERE TABLESPACE_NAME='FUSION';
8
Verify all datafiles were added with unique names
SELECT FILE_NAME, TABLESPACE_NAME
FROM DBA_DATA_FILES
WHERE TABLESPACE_NAME = 'FUSION'
ORDER BY FILE_NAME;
Lists all datafiles for FUSION tablespace. Each has a unique ASM path. This confirms Oracle ASM is managing them distinctly.
9
Enable autoextend on a tablespace datafile
SELECT FILE_NAME FROM DBA_DATA_FILES
WHERE TABLESPACE_NAME='FUSION' AND ROWNUM=1;

ALTER DATABASE DATAFILE '+DATA01/DBNAME/DATAFILE/fusion.344.1164235091'
AUTOEXTEND ON NEXT 100M MAXSIZE 2G;
Database altered. The datafile will now grow automatically up to 2GB. Always set a MAXSIZE to prevent runaway growth.
4

Common Errors & Fixes

Error / SymptomWhat to Do
ORA-15041: diskgroup space exhaustedThe ASM diskgroup is full. Check: SELECT NAME,FREE_MB FROM V$ASM_DISKGROUP; Need to either clean up old files or have the storage team add disks.
ORA-01543: tablespace already existsA tablespace with that name already exists. Choose a different name or check: SELECT TABLESPACE_NAME FROM DBA_TABLESPACES;
ORA-01119: error in creating database fileThe ASM diskgroup name is wrong. Check: SELECT NAME FROM V$ASM_DISKGROUP; and use the exact name (e.g., +DATA01 not +data01).
Autoextend reaches MAXSIZE — tablespace fills upAdd a new datafile: ALTER TABLESPACE fusion ADD DATAFILE '+DATA01' SIZE 500M AUTOEXTEND ON NEXT 100M MAXSIZE 5G;

📜 Execute SQL Scripts in a Database

The correct workflow for running a SQL script safely in production

Skill Level
Beginner
Time Required
15 min
Category
Database Administration
Prerequisites
oracle OS user · Script file accessible on the server · sqlplus available
Applies To
Oracle 19c / 21c
1

What You Will Learn

Running SQL scripts is a core DBA task — whether deploying application changes, running DDL, or executing patches. Doing it correctly means reviewing the script first, connecting as the right user, spooling output to a log file, and always verifying the results. This guide walks through the complete workflow.
2

Before You Begin

  • Have the script file on the database server
    Use scp or a shared folder to transfer. Confirm: ls -l scriptname.sql
  • Know which database and user to run it as
    The script's comments should specify this. Never run an app script as SYSDBA unless it explicitly requires it.
  • Review the script before running it
    vi scriptname.sql or cat scriptname.sql — check for DROP statements, dangerous DDL, or anything unexpected
3

Step-by-Step Instructions

#What to DoCommandExpected Result
1
Review the script using vi
vi unidb.sql
# Navigation: arrow keys, / to search
# Quit: :q Quit without saving: :q!
Review every SQL statement. Look for DROP TABLE, TRUNCATE, DELETE without WHERE, or anything that could cause data loss. Never run an unreviewed script.
2
Set the correct database environment
. oraenv
# At prompt, enter the target SID:
ORACLE_SID = [kud04] ? kud04
The Oracle base and HOME are configured. Confirm: echo $ORACLE_SID should return your target database.
3
Navigate to the directory containing the script
cd /home/oracle/scripts/
pwd
ls -l *.sql
pwd confirms your location. ls verifies the script file exists and is readable. Correct permissions: -rw-r--r-- oracle oinstall
4
Start sqlplus and verify the target database
sqlplus / as sysdba
SELECT NAME FROM V$DATABASE;
The database name must match your expected target. ALWAYS verify this before running any script — wrong database = disaster.
5
Connect as the correct application account
CONNECT fusion
-- Enter password at prompt
SHOW USER
USER is "FUSION". Application scripts must run as the schema owner, not as SYSDBA (unless the script requires it).
6
Enable spooling to capture all output
SPOOL unidb.log
Spooling started. All SQL output, errors, and results will be captured in unidb.log in the current directory.
7
Execute the script
@unidb.sql
Tables created/dropped, rows inserted, etc. Watch for any ORA- error messages. If you see errors, type SPOOL OFF immediately and review the log.
8
Turn off spool and exit sqlplus
SPOOL OFF
EXIT
Spool file is closed. You are back at the OS prompt.
9
Review the log file for errors
vi unidb.log
# Or:
grep -i "ORA-\|error" unidb.log
If grep returns no results — no errors detected. If errors exist, review them and determine if the script completed its intent or if a rollback/rerun is needed.
10
Verify the results in the database
sqlplus / as sysdba
SELECT OBJECT_TYPE, OBJECT_NAME, STATUS
FROM DBA_OBJECTS
WHERE OWNER = 'FUSION'
ORDER BY OBJECT_TYPE;
Confirms what objects were created. STATUS should be VALID for all. Any INVALID objects need investigation.
4

Common Errors & Fixes

Error / SymptomWhat to Do
ORA-01017 when connecting as application userWrong password or user does not exist. Check: SELECT USERNAME FROM DBA_USERS WHERE USERNAME='FUSION';
Script runs but spool file is emptyVerify SPOOL command was run BEFORE @script.sql. The SPOOL command must precede script execution.
ORA-00942: table or view does not exist during scriptScript may be running as wrong user, or dependent objects do not exist. Check the script's prerequisites and connection user.
Cannot find the spool log file after runningThe spool file is created in the current working directory at the time sqlplus was started. Check: from OS run: find /home/oracle -name "*.log" -newer /tmp -ls

🗄️ Configure Archivelog Mode

Enable or disable archivelog mode — required for online backups and recovery

Skill Level
Intermediate
Time Required
20 min (+ restart downtime)
Category
RMAN Backup & Recovery
Prerequisites
SYSDBA · DB can be restarted · FRA (Fast Recovery Area) configured
Applies To
Oracle 19c / 21c
1

What You Will Learn

Archivelog mode is a database-level setting that determines whether Oracle saves copies of filled redo log files (archivelogs). Without archivelog mode you cannot do online backups or point-in-time recovery. Every production Oracle database should be in archivelog mode. This guide walks through checking the mode, enabling it, and verifying archivelogs are being written.
2

Before You Begin

  • You have a maintenance window — enabling archivelog requires a restart
    Coordinate with the application team before shutting down
  • Configure the Fast Recovery Area first
    SET db_recovery_file_dest and db_recovery_file_dest_size before enabling archivelog mode
  • Verify the FRA diskgroup has enough space

🧹 Clean Up the FRA When It Is Full

Emergency procedure for when the Fast Recovery Area reaches 100% capacity

Skill Level
Intermediate
Time Required
20 min
Category
RMAN Backup & Recovery
Prerequisites
RMAN access · SYSDBA · Alert log showing ORA-19809 or database hanging
Applies To
Oracle 19c / 21c with ASM or filesystem FRA
1

What You Will Learn

When the FRA fills to 100%, the database stops writing archivelogs and hangs — all transactions wait and the application goes down. This is a DBA emergency. This guide walks through the exact procedure: identifying what is consuming space, safely deleting old archivelogs and obsolete backups, and verifying the database resumes normally. Prevention is covered at the end.
2

Before You Begin

  • Confirm the FRA is actually the problem
    Check alert log: grep "ORA-19809\|archiv" $ORACLE_BASE/.../alert_SID.log | tail -20
  • Do NOT delete any files manually from the OS
    Only use RMAN to delete backup files — manually deleting them confuses RMAN and causes catalog corruption
  • Have RMAN credentials ready
    rman target / connects locally. For RMAN catalog: rman target / catalog rman_user/pass@catalog
3

Step-by-Step Instructions

#What to DoCommandExpected Result
1
Check FRA usage to confirm it is the issue
SELECT
ROUND(SPACE_USED/SPACE_LIMIT*100,1) pct_used
FROM V$RECOVERY_FILE_DEST;

SELECT NAME, FREE_MB, TOTAL_MB FROM V$ASM_DISKGROUP;
If PCT_USED = 100 or FRA01 FREE_MB is near 0, FRA is the problem. This confirms your diagnosis.
2
Identify which file types are consuming the most space
SELECT FILE_TYPE, PERCENT_SPACE_USED, NUMBER_OF_FILES
FROM V$FLASH_RECOVERY_AREA_USAGE
WHERE PERCENT_SPACE_USED > 0
ORDER BY PERCENT_SPACE_USED DESC;
Archivelogs are usually the biggest consumer. Flashback logs indicate a restore point is set. Backup pieces indicate old backups.
3
Check for restore points holding flashback logs
SELECT NAME, TIME, STORAGE_SIZE/1024/1024/1024 size_gb
FROM V$RESTORE_POINT;
A guaranteed restore point prevents flashback log deletion. If the restore point is no longer needed, drop it — this releases the most space quickly.
4
Drop restore point if no longer needed
-- ONLY if you confirm it is not needed:
DROP RESTORE POINT grp_name;

-- Verify space reclaimed:
SELECT NAME, FREE_MB FROM V$ASM_DISKGROUP;
Space is immediately reclaimed after dropping the restore point. Verify FREE_MB increased on the FRA diskgroup.
5
Delete old archivelogs via RMAN
rman target /

DELETE NOPROMPT ARCHIVELOG
ALL COMPLETED BEFORE 'SYSDATE-2';

EXIT;
Deletes all archivelogs older than 2 days. RMAN updates its catalog, so no orphaned records remain. Adjust the 2-day threshold based on your backup retention policy.
6
Delete obsolete backups
rman target /

DELETE NOPROMPT OBSOLETE;

EXIT;
Removes backups no longer needed per the retention policy. This is safe — RMAN only deletes what it knows is not needed for recovery.
7
Verify space has been reclaimed
SELECT
ROUND(SPACE_USED/SPACE_LIMIT*100,1) pct_used
FROM V$RECOVERY_FILE_DEST;

SELECT NAME, FREE_MB, TOTAL_MB FROM V$ASM_DISKGROUP;
PCT_USED should be significantly lower. FRA01 FREE_MB should be much higher. Target: below 75% used.
8
Verify the database has resumed archiving
SELECT SEQUENCE#, NAME, COMPLETION_TIME
FROM V$ARCHIVED_LOG
WHERE COMPLETION_TIME > SYSDATE - 1/24
ORDER BY SEQUENCE# DESC;
New archivelogs should be appearing with recent timestamps. If no new entries, the database may still be hung — check the alert log.
9
Check the alert log for recovery
tail -50 $ORACLE_BASE/diag/rdbms/$(echo $ORACLE_SID | tr A-Z a-z)/$ORACLE_SID/trace/alert_${ORACLE_SID}.log
Look for "Archiver continuing" or "LGWR: STARTING ARCH PROCESSES" messages. ORA-19809 errors should stop appearing.
4

Common Errors & Fixes

Error / SymptomWhat to Do
RMAN DELETE hangs or is very slowThe database is under heavy archiving pressure. Wait for DELETE to complete — interrupting it can leave the catalog inconsistent.
Space reclaimed but FRA still shows 100%V$RECOVERY_FILE_DEST may cache the old value. Query V$ASM_DISKGROUP instead for the real-time physical space view.
Archivelogs keep accumulating — database fills up againThe backup job is not deleting archivelogs after backing them up. Check your RMAN script: BACKUP ... DELETE INPUT or ADD DELETE ARCHIVELOG in the channel config.
Cannot drop restore point — ORA-38778The restore point is set with GUARANTEE FLASHBACK DATABASE and is within the flashback retention window. You may still drop it: DROP RESTORE POINT name; — this is safe to do.

Oracle 19c / 21c
1

What You Will Learn

A restore point is a named bookmark in the database timeline. Before any risky operation — a patch, a release deployment, a data load — you create a GUARANTEED restore point. If something goes wrong, you can instantly flashback the entire database to that exact moment. This is Oracle's equivalent of a "system checkpoint" before making changes.
2

Before You Begin

  • Flashback database must be enabled
    SELECT FLASHBACK_ON FROM V$DATABASE; — must be YES
  • FRA has sufficient free space
    A guaranteed restore point forces Oracle to keep flashback logs — this consumes FRA space
  • Know the name of the restore point you will create
    Use a descriptive name: BEFORE_PATCH_JAN2025, PRE_RELEASE_V3, BEFORE_DATA_LOAD
3

Step-by-Step Instructions

#What to DoCommandExpected Result
1
Check if any restore points already exist
SELECT NAME, TIME,
GUARANTEE_FLASHBACK_DATABASE,
STORAGE_SIZE/1024/1024/1024 size_gb
FROM V$RESTORE_POINT;
No rows = no restore points. If rows exist, review them — old guaranteed restore points consume FRA space and should be dropped when no longer needed.
2
Create a guaranteed restore point
CREATE RESTORE POINT before_change
GUARANTEE FLASHBACK DATABASE;
Restore point created. GUARANTEE FLASHBACK DATABASE means Oracle guarantees it can flash back to this point regardless of archivelog deletion — it keeps the necessary logs.
3
Verify the restore point was created
SELECT NAME,
TO_CHAR(TIME,'DD-MON-YY HH24:MI') created_at,
GUARANTEE_FLASHBACK_DATABASE
FROM V$RESTORE_POINT;
Shows: BEFORE_CHANGE with the creation timestamp and GUA=YES. You can now proceed with your risky operation knowing you can rewind.
4
Make your risky change (example: deploy a patch)
-- Proceed with whatever operation you are protecting:
-- Apply patch, run data load, execute release scripts, etc.
Document what you did. The restore point captures the state BEFORE your changes.
5
If something goes wrong: flashback to the restore point
-- Shut down the database:
SHUTDOWN IMMEDIATE;
-- Mount (not open):
STARTUP MOUNT;
-- Flashback:
FLASHBACK DATABASE TO RESTORE POINT before_change;
-- Open with resetlogs:
ALTER DATABASE OPEN RESETLOGS;
Database flashed back and open. All changes since the restore point are gone. Verify: SELECT NAME, OPEN_MODE FROM V$DATABASE;
6
If everything went fine: drop the restore point
DROP RESTORE POINT before_change;
Restore point dropped. This releases the FRA space that was held to guarantee the flashback. Always drop restore points when they are no longer needed.
7
Verify restore point is dropped and space reclaimed
SELECT NAME FROM V$RESTORE_POINT;
SELECT NAME, FREE_MB FROM V$ASM_DISKGROUP;
No rows in V$RESTORE_POINT. FRA diskgroup FREE_MB should have increased.
4

1

What You Will Learn

Flashback Database is Oracle's fastest recovery mechanism. Unlike RMAN restore (which can take hours), flashback can rewind a database in minutes by applying reverse-change vectors from flashback logs. This guide walks through the complete procedure: confirming the restore point, shutting down, flashing back, and reopening with RESETLOGS.
2

Before You Begin

  • Confirm the restore point still exists
    SELECT NAME, TIME FROM V$RESTORE_POINT; — if no rows, you cannot use this procedure. Use RMAN restore instead.
  • Notify all users — the database will be unavailable
    Flashback requires mounting the database, so all connections will be lost
  • This will undo ALL changes since the restore point was created
    Coordinate with the application team to identify what will be lost and whether that is acceptable
3

Step-by-Step Instructions

#What to DoCommandExpected Result
1
Confirm flashback is enabled and restore point exists
SELECT NAME, OPEN_MODE, FLASHBACK_ON FROM V$DATABASE;
SELECT NAME, GUARANTEE_FLASHBACK_DATABASE,
TO_CHAR(TIME,'DD-MON-YY HH24:MI') created
FROM V$RESTORE_POINT;

2
Note the current SCN for reference
SELECT CURRENT_SCN FROM V$DATABASE;
Record this number. It is the "now" point. After flashback, you will be rewound to before this SCN.
3
Shut down the database
SHUTDOWN IMMEDIATE;
Database closed. Database dismounted. ORACLE instance shut down.
4
Start the database in MOUNT mode
STARTUP MOUNT;
ORACLE instance started. Total System Global Area XXXX bytes. Database mounted. Do NOT open it yet — flashback requires MOUNT state.
5
Execute the flashback
FLASHBACK DATABASE TO RESTORE POINT before_change;
Flashback complete. Oracle applied reverse flashback logs to rewind all datafiles to the restore point's SCN. This typically takes 2-10 minutes.
6
Open the database with RESETLOGS
ALTER DATABASE OPEN RESETLOGS;
Database altered. RESETLOGS creates a new database incarnation. This is required after any incomplete recovery or flashback.
7
Open all PDBs if this is a CDB
ALTER PLUGGABLE DATABASE ALL OPEN;
All PDBs opened. In a non-CDB environment, skip this step.
8
Verify the database is open and healthy
SELECT NAME, OPEN_MODE, LOG_MODE FROM V$DATABASE;
SELECT COMP_NAME, STATUS FROM DBA_REGISTRY
WHERE STATUS != 'VALID';
OPEN_MODE = READ WRITE. Zero invalid components. Database is back to the pre-change state.
9
Take an immediate backup
rman target /
BACKUP DATABASE PLUS ARCHIVELOG DELETE INPUT;
EXIT;
Critical: after RESETLOGS, previous backups cannot be used for incremental recovery. Start fresh with a new backup.
10
Drop the restore point now that it has been used
DROP RESTORE POINT before_change;
Restore point dropped. This releases the FRA space that was reserved for the guarantee.
4

HOW-TO GUIDE

🩺 Troubleshoot ORA-15041: FRA Diskgroup Space Exhausted

Step-by-step investigation and fix when the shared FRA ASM diskgroup fills up

Skill Level
Advanced
Time Required
30–60 min
Category
RMAN Backup & Recovery
Prerequisites
Access to all databases on the server · SYSDBA · RMAN · ASM knowledge
Applies To
Oracle 19c / 21c with shared ASM FRA diskgroup
1

What You Will Learn

ORA-15041 means an ASM diskgroup has run out of space. When the FRA diskgroup is exhausted, ALL databases on the server are affected — not just one. Flashback cannot start, archivelogs cannot be written, and databases may hang. This guide walks through identifying which databases are using the most space, finding and dropping old restore points, and deleting old archivelogs to reclaim space.
2

Before You Begin

  • This impacts ALL databases sharing the FRA diskgroup
    You may need to investigate multiple databases on the server
  • Do NOT manually delete files from the ASM filesystem
    Only use RMAN or SQL commands — manual deletion corrupts the RMAN catalog
  • Have a list of all databases on the server
    cat /etc/oratab — shows all registered SIDs
3

Step-by-Step Instructions

#What to DoCommandExpected Result
1
Identify the error and confirm FRA diskgroup is full
SELECT NAME, FREE_MB, TOTAL_MB,
ROUND(FREE_MB/TOTAL_MB*100,2) pct_free
FROM V$ASM_DISKGROUP;
FRA01 FREE_MB near 0 confirms the diskgroup is full. DATA01 may still have space — this is a FRA-specific issue.
2
Check FRA file type breakdown for current database
SELECT FILE_TYPE,
PERCENT_SPACE_USED,
PERCENT_SPACE_RECLAIMABLE,
NUMBER_OF_FILES
FROM V$FLASH_RECOVERY_AREA_USAGE
WHERE PERCENT_SPACE_USED > 0
ORDER BY PERCENT_SPACE_USED DESC;
Identifies which file types are consuming space. Usually: ARCHIVED LOG or FLASHBACK LOG at the top.
3
Check for restore points holding flashback logs
SELECT NAME, TIME,
STORAGE_SIZE/1024/1024/1024 size_gb
FROM V$RESTORE_POINT;
A guaranteed restore point forces Oracle to keep flashback logs. If the restore point is old and no longer needed, dropping it reclaims significant space immediately.
4
Drop old restore point if confirmed not needed
-- Only after confirming with app team it is not needed:
DROP RESTORE POINT grp_name;

SELECT NAME, FREE_MB FROM V$ASM_DISKGROUP;
Check FREE_MB after dropping. Often reclaims 1-5 GB immediately.
5
Switch to each database and check its FRA usage
-- Repeat for each database on the server:
. oraenv # enter next DB SID
sqlplus / as sysdba
SELECT FILE_TYPE, PERCENT_SPACE_USED
FROM V$FLASH_RECOVERY_AREA_USAGE
WHERE PERCENT_SPACE_USED > 5
ORDER BY PERCENT_SPACE_USED DESC;
Identifies which databases are the biggest consumers. The database using 44% archivelogs + 20% flashback logs is the primary target.
6
Delete old archivelogs via RMAN for the biggest consumer
rman target /

DELETE NOPROMPT ARCHIVELOG
ALL COMPLETED BEFORE 'SYSDATE-2';

EXIT;
RMAN deletes archivelogs older than 2 days and updates its catalog. Check V$ASM_DISKGROUP after this step.
7
Monitor space recovery
SELECT NAME, FREE_MB, TOTAL_MB,
ROUND(FREE_MB/TOTAL_MB*100,1) pct_free
FROM V$ASM_DISKGROUP;
FREE_MB should be significantly higher. Target: get to at least 10-20% free space.
8
Retry the original failed operation
-- If flashback was failing:
ALTER DATABASE FLASHBACK ON;
SELECT FLASHBACK_ON FROM V$DATABASE;
FLASHBACK_ON = YES confirms the root cause is resolved. The space was the problem.
4

Common Errors & Fixes

Error / SymptomWhat to Do
Dropping restore point reclaims no spaceThe flashback logs for that restore point may have already been overwritten. Check STORAGE_SIZE in V$RESTORE_POINT — very small = logs already gone.
RMAN DELETE is very slowLarge number of archivelogs. Let it run — do not interrupt. Monitor progress: SELECT STATUS FROM V$RMAN_STATUS ORDER BY START_TIME DESC;
Space freed but FRA fills up again quicklyArchivelog generation rate exceeds backup/deletion frequency. Investigate archivelog generation rate: SELECT COUNT(*), TRUNC(COMPLETION_TIME,'HH') FROM V$ARCHIVED_LOG GROUP BY TRUNC(COMPLETION_TIME,'HH') ORDER BY 2 DESC;
Cannot connect to one of the databases to checkIf a database is hung due to full FRA, it may reject connections. Free space from other databases first, then the hung database should resume.
